#CouchDB
[원본 링크]

CouchDB는 2005년에 만들어진 대규모 분산 DB를 위한 시스템이다.
Erlang으로 만들어졌고, Apache에 의해 관리되며, Document 기반의 NoSQL이다.




특징


https://www.javatpoint.com/couchdb-tutorial
MongoDB와 비슷하게 JSON Document를 저장 형식으로 사용한다.

그리고 multi master 기반의 레플리케이션을 기본으로 사용하며, 높은 가용성과 강력한 최종 일관성 보장을 주요 장점으로 내세운다.

NoSQL으로서는 독특하게 LSM Tree가 아닌 B-Tree를 기본 인덱스 매커니즘으로 사용한다.

query 방식은 map-reduce 방식을 주로 사용한다.

특이하게 DB로서 자체 프로토콜을 사용하지 않고 HTTP/REST 통신으로 모든 것을 처리한다. 그래서 조금 느릴 수도 있다.





vs MongoDB

Document DB라는 공통점 때문에 MongoDB와 곧잘 비교되곤 하는데, 차이점이 꽤 많은 편이다.

1. 복제
MongoDB는 single master - multi slave 구조로 복제를 처리하는 반면, Couch는 multi master로 복제를 처리한다. 이러한 구조 탓에 가용성 측면에서는 Couch가 더 유리하다.

2. 동시 제어
Couch는 타임스탬프 기반으로 동시 제어를 수행하고 낙관적인 Lock 기법을 사용한다. 그래서 write가 read를 차단하지 않는다.
만약 여러개의 명령이 동시에 들어와서 충돌이 발생한다면 가장 먼저 들어온 것을 유지하고 나중에 들어온 것을 버리는 방식을 취한다. 그래서 Application 수준에서 충돌을 처리하는게 더 복잡할 수도 있다.

3. 트랜잭션
MongoDB는 불완전하게나마 트랜잭션 기능을 제공하나, Couch는 그런게 딱히 없다.

4. 성능
일반적인 읽기 성능이나 대규모 데이터를 다루는 능력은 MongoDB가 Couch보다는 나은 편이다.

5. 인덱스
MongoDB는 RDB 수준의 다양한 인덱스 기능이 지원되나, Couch는 잘 지원되지 않는다.

6. query
MongoDB는 자체적인 json 형식의 query를 주로 사용하나, couch는 HTTP Request를 통해 query를 날린다.

전반적인 편의성에서는 MongoDB가 더 적합한 편이다.
Couch는 일반적인 비즈니스 서비스의 메인 DB로는 사용하는데 한계가 있다.




vs Couchbase

Couch 환경을 접할때 가장 혼란스러운 것 중 하나가 Couch* 시리즈의 괴상한 아종들이 굉장히 많다는 것이다.

그 중 대표적인 것이 Couchbase인데, 좀 다른 물건이지만 보통 CouchDB의 엔터프라이즈 버전이라고 인식이 되는 편이다.
하지만 완전히 별개의 프로젝트고, 구조도 다르다. 일부 역사를 공유할 뿐이다.

역사를 대충 정리하자면, CouchDB를 만들었던 회사가 Memcached를 만든 회사와 합착해서 Membase라는 것을 만들었는데, 그게 Couchbase가 된 것이다. 그래서 이건 프로토콜도 Memcached의 것을 사용한다.
