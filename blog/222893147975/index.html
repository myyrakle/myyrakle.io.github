[Java] Java16: Record Class

  • 2022.10.06
  • ::
  • 1 min read
#Java
[원본 링크]

자바 하면 안티패턴으로 순위에 꼽히는 것 중 하나가 바로 Getter Setter 패턴이다.

이 패턴은 단순한 데이터성 타입(DTO 등)을 다룰 때 사용하곤 한다.
그냥 name과 age 값 2개만 갖는 데이터 타입을 자바스타일로 짜면 저렇게 된다.
당연히 필드가 늘어나면 늘어날수록 코드라인도 급격하게 늘어난다.

저게 의미가 있어 보이나?
아무도 그 이유를 모르면서 쓰는 적폐 패턴중 하나다.
말로는 쓰면서 캡슐화니 은닉화니 객체지향이니 하는데, 저게 어딜봐서 객체지향적이란 건가?
애초에 다 get set 만들거면 은닉화가 되지도 않는다.


그나마 최신 버전에서는 이러한 구태들을 벗어나려는 노력들을 조금 하긴 하는데, 효과는 없을 것 같다.

아무튼 record class는 이런 데이터성 타입들을 정의할 때 사용하라고 만든 특수한 클래스 정의법이다.
신택스는 대강 다음과 같은 형태를 가진다.

record 레코드명 (필드1, 필드2, ...)
{
   ... 클래스 정의
}

레코드는 생성자를 통해 불변 객체를 생성하며, 필드명과 동일한 메서드를 통해 필드값을 조회할 수 있다.
원한다면 이런저런 메서드들을 직접 구현해서 사용할 수도 있다.

예시코드다.

public class MyClass {
    public static void main(String args[]) {
        var john = new Person("john", 155);

        System.out.println("name: "+john.name());
        System.out.println("age: "+john.age());
    }
}

record Person (String name, int age)
{ }

저기서 정의한 Person 레코드는

record Person (String name, int age)
{ }

실제로 이렇게 변환이 된다.

public final class Person {
    private final String name;
    private final int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    String name()  { return this.name; }
    int age() { return this.age; }

    public boolean equals...
    public int hashCode...

    public String toString() {...}
}
