DI 시스템을 직접 구현하면서 any 떡칠을 좀 해보게 되었는데, 거기서 얻은 사용상의 주의점을 대략 정리해보겠다.




기본 사용법

any의 기본적인 사용법은 별거 없다.
any 트레잇으로 박싱된 객체를 생성하면, 그것 자체로 any 타입의 동적 객체가 되는 것이다.

그리고 downcast를 통해서 원래의 값을 타입 안전하게 가져올 수 있다.




TypeId

근데 보통 저것만으로는 부족하다.

보통 any 떡칠을 해야할 상황이 오면 타입정보를 들고서 그때그때 그걸 기반으로 뭔가 처리를 해야할 때가 많은데, 이걸 위한 보조기능이 TypeId다.

TypeId::of 제너릭 함수를 사용하면 해당 타입에 대한 타입 고유번호를 가져올 수 있다.

이런식으로 하나의 타입마다 고유한 값이 발급된다.

이건 제너릭이 끼어들어도 크게 달라지진 않는다.
실제로 동등한 타입인지를 기준으로 Id를 발급하기 때문에, 같은 타입으로 여러번 제너릭을 구체화하더라도, 타입이 호환되면 같은 TypeId가 나온다.




Box와 TypeId

근데 Any Box을 해서 쓰다보면 좀 혼동할 수 있는 부분이 있다.
이런식으로 박싱을 하고, 그냥 기존 타입과 동등할 것이라 가정하고 쓰면

TypeId가 다르게 나온다.
저게 왜나면, 현재 타입을 기준으로 Id를 발급하는 것인데 저 boxed는 Box<>로 감싸인 또 별개의 타입이기 때문이다.

만약 박싱된 Any 오브젝트에서 진짜 타입값을 꺼내고 싶다면 역참조를 해서 언박싱된 dyn Any 참조를 가져와야 한다.




트레잇 any 박싱하기

만약 이미 박싱된 트레잇 객체를 any로 만들고 싶다면 어떻게 해야할까?
사실 방법은 다를게 없다.

이중박싱을 하면 된다.
트레잇 박싱을 먼저 하고, any로 한번 더 박싱을 한다.

any 다운캐스트로 꺼내올때도 트레잇 박스로 가져오면 된다.
