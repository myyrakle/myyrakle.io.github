[AWS] VPC 설정하기

  • 2021.04.21
  • ::
  • 5 min read
#AWS
[원본 링크]

VPC는 Virtual Private Cloud의 준말로, 가상의 네트워크 망을 구축해주는 기능이다.

이게 왜 필요할까?
AWS로 서비스를 구현하다보면, 당연히 AWS 내의 서비스들끼리 통신을 해야 할 경우가 잦다.
EC2에서 RDS로 통신을 쏘고, 그 EC2에서 다른 EC2로 요청을 쏘고, 어떤 Lambda는 EC2들에 전부 요청을 쏘고...

이러다보면 서로 다 아이피를 허용해주고 그래야 하니 여러모로 불편한 것이 많다.

하지만 VPC로 네트워크망을 묶어버리는 이에 대한 처리와 관리가 아주 간편해지는 것이다.

뿐만 아니라 VPC를 설정해둔 서비스는 해당 VPC 안에서만 쓸수 있기 때문에, 보안적 측면에서도 아주 안정적이다.

한번 들어가서 설정을 해보자.
별다른 설정을 주지 않았더라도 VPC와 보안그룹이 이렇게 한개씩은 깔려있을 것이다.

이름이 -이라 식별이 안되니 알아볼 수 있게 바꿔주자.

이름을 클릭하면 수정 창이 뜬다.

서브넷도 바꿔준다. 서브넷은 VPC 내에 N개 이상 존재할 수 있다.
VPC가 그냥 논리적인 범위일 뿐이라면, 실제로 리소스가 생성될 수 있는 네트워크다.

라우팅 테이블도 바꿔주고

인터넷 게이트웨이

DHCP 옵션도 바꿔준다.

보안그룹도 마찬가지다. 이건 서비스 앞단에서 방화벽 역할을 한다.

그리고 네트워크 ACL도 바꿔준다. 이건 서브넷 앞단에서 일종의 방화벽 역할을 하는 친구다.

이제 저건 냅두고 이제 제대로 사용할 것들을 만들어보겠다.




탄력적 IP

탄력적 IP는 이름은 거창해보이는데, 그냥 고정 IP를 할당해주는 것을 말한다.

그냥 기본값으로 할당해주면 된다.

EC2에 연결을 해두는게 아니면 돈도 조금 나가니 유의한다.



키 페어 생성

이번엔 EC2->키 페어로 들어가서 생성하고, 비밀키 .pem을 잘 보관해둔다.

이 키값은 한번 뽑으면 다시는 안주니까, 유실되지 않도록 잘 보관해야 한다.




VPC 생성

이제 밑작업을 대강 마쳤으니, 본격적으로 사용할 VPC를 만들어보자.

RDS 데이터베이스에 사용할 환경을 구축한다고 가정하겠다.

대시보드에서 VPC 마법사를 클릭한다.

그냥 VPC 탭에서 생성을 해도 되지만, 이걸로 하는게 더 편리하다.

퍼블릭 및 프라이빗 서브넷을 선택하고

2단계에선 아래와 같이 설정한다.

이름을 지어주고, 가용영역을 a인 것으로 설정, 방금 만든 탄력적 IP를 붙여준다.

완료를 누르면, 생성이 5분 정도 걸린다.




서브넷 생성

이제 VPC 아래에다 사용할 서브넷을 생성해주자.
이게 실질적으로 사용할 네트워크 영역이다.

RDS용으로 사용할 서브넷을 하나 만들도록 하겠다.
vpc를 선택하고, 이름을 잘 짓고, 가용영역 선택해주고, CIDR 영역을 잘 지정해준다.
CIDR은 VPC 영역 내에서 범위를 어떻게 가져갈 지를 지정해주는 값이다. 다른 항목과 겹치지만 않고, 범위를 벗어나지 않으면 된다.

완료




라우팅 테이블 생성

라우팅 테이블은 서브넷에서 네트워킹을 할때 사용하는 목적지에 대한 테이블이다.
하나의 라우팅 테이블은 여러개의 서브넷과 연결될 수 있다.
다음과 같이 VPC를 지정해서 생성한 후에

하나 이상의 서브넷과 연결할 수 있다.

나는 일단 a 가용영역의 서브넷에만 연결했는데,

장애상황을 고려한다면 c와 d 가용영역에 대한 서브넷을 생성해서 추가로 연결할 수 있다.
그렇게 한다면 a 서브넷에 문제가 생기더라도 c와 d에서 그 역할을 대신할 수 있다.




다시 이름 정리

VPC 마법사로 VPC를 생성했다면, 보안그룹이나 이런저런 것들은 또 자동으로 만들어줬을 것이다.
하지만 이름을 이전과 동일하게 -로 깔아버리므로... 보기 좋게 다시 바꿔주자.
보안 그룹 등을 위에서 했던 것과 동일하게 잘 지어주면 된다.




RDS-서브넷 그룹 생성

데이터베이스에 사용할 서브넷 그룹을 생성한다. 이걸로 RDS가 어디에 놓일지를 정의할 것이다.

이름 잘 짓고, VPC 고르고, 가용영역과 서브넷을 선택한다.

여기서 특이한 부분으로, RDS는 안정성을 위해서 2개 이상의 가용영역을 지정해야 한다.
그래야 하나가 만일의 사고로 죽더라도 다른 영역에서 계속 동작을 보장할 수 있기 때문이다.

그래서 c로 서브넷 하나 더 만들어서 지정해줬다.




보안 그룹 생성

그리고 공통적으로 사용할 보안 그룹을 하나 생성한다.
VPC를 지정하고 공개할 형태를 [인바운드 규칙]에서 추가한다.

일반적으로 HTTP와 HTTPS는 다 열어두고, SSH는 어떤짓을 할지 모르니 아이피를 제한시켜둔다.

그리고 디비 커넥션을 허용할 대상을 적절히 지정해준다.
만약 람다에서 커넥션을 맺어야 한다면, 그 람다가 가질 보안그룹 등을 IP 항목에 지정해주면 된다.




RDS에 VPC 설정하기

다른 부분은 차치하고 VPC를 설정하는 부분만 보겠다.
어려울 것은 없다. 먼저 VPC를 선택하고 방금 만든 서브넷 그룹이 들어가도록 하면 끝이다.

이외의 RDS 설정들에 대한 것들은 RDS 포스트에서 다시 정리하겠다.





람다로 RDS 접속하기

이제 거의 다 왔다.
람다를 만들되, VPC와 서브넷, 보안그룹을 적용해서 만들어주면 된다.

여기서 지정한 보안그룹이 Postgres에서 사용한 보안그룹-인바운드 규칙에 있어야 한다.

그리고 뭐... 레이어로 데이터베이스 커넥션 라이브러리 올리고, 대충 돌려보면 될 것이다.

이렇게.
데이터베이스 도메인은 RDS 엔트리포인트를 긁어오면 된다.

그럼 잘 될 것이다.

그렇다.
