단순히 API에 대한 기록을 보고 싶다면 Tempo 등을 사용해서 trace를 기록하면 된다.
그러면 API들이 얼마나 호출되었는지, 오류율이 얼마인지 같은 통계성 데이터를 보고 싶다면 어떻게 해야할까?

그런걸 위한 기능요소 중 하나가 바로 spanmetric이다. 각각의 span을 기반으로 metric을 구성한다는 의미다.

본 포스트에서는 otel-collector를 기반으로 prometheus에 통계를 쑤셔넣고 사용하는 방법을 간단히 다뤄보겠다.




otel-collector 설정

spanmetric은 실제로 span-trace 데이터를 기준으로 생성되는 메트릭 데이터다.

connectos 옵션을 통해서 데이터를 어떻게 넘겨받을 지를 정의할 수 있다.

connectors:
  spanmetrics:
    namespace: http_server_request
    dimensions:
      - name: http.method
      - name: http.status_code
      - name: http.route
      - name: service.namespace	
    histogram:
      unit: s
    exemplars:
      enabled: true
    events:
      enabled: true
      dimensions:
        - name: exception.type
        - name: exception.message

http_server_request라는 접두어로 metric이 쌓이게 했고, http.method를 비롯한 span의 데이터들을 계승받아서 저장하도록 했다.
그리고 기록 단위는 초 단위다. 기본값은 밀리초단위인데, 난 그정도는 필요없어서 이렇게 했다.

이 부분이 특히 grafana 대시보드 환경에서 특히 개떡같이 되어있는 부분 중 하나다.
보통은 대시보드 템플릿 라이브러리를 가져다가 편하게 쓰는 것을 원할 것이다.

로그나 trace는 어느 정도 정형화가 되어있고 선형적인 데이터라서 대충 쑤셔넣고 아무거나 띄우면 시각화가 잘 되는데, 이건 잘 안된다.
otel-collector 자체가 하위호환을 보장하지 않고 0,* 상태로 폭주중인 것도 있고, span에게서 계승받는 데이터도 다 자기들 맘대로 채워넣고 그런다. 그래서 grafana 라이브러리에 있는 opentelemetry 템플릿들 보면 데이터 형태나 metric 이름을 가정한게 다 다르다.

제대로 쓰려면 어느 정도 데이터 수집 구조를 알고 설정할 줄도 알아야 한다.

service에도 spanmetric 관련 옵션을 적당히 채워준다.

service:
  pipelines:
    metrics:
      receivers: [otlp]
      exporters: [prometheus]
    metrics/spanmetrics:
      receivers: [spanmetrics]
      exporters: [prometheus]
    traces:
      receivers: [otlp]
      exporters: [spanmetrics, otlp]
  telemetry:
    metrics:
      address: 0.0.0.0:8888
      level: detailed

이 정도면 된다.

span->metric 전이는 otel-collector 수준에서 다 해주는거라서, 이미 metric&trace 설정이 잘 되어있다면 애플리케이션 수준에서 에이전트 조정을 더 해줘야 할 것은 없다.

그런데 job이 좀 이상하다. 이 경우에는 k8s에서의 사용만을 전제하고 저 필드가 있을 것이라 기대하는데, 지금 내 환경과 구성에서는 service_namespace와 service_name이 따로 있다.
