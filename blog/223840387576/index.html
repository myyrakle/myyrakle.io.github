Vector Search 벤치마크 (pgvector, elasticsearch, qdrant)

  • 2025.04.20
  • ::
  • 4 min read
#데이터베이스
[원본 링크]

Vector Search를 약간 실험적으로 도입해볼 필요가 있었는데, 리소스를 대략적으로 얼마나 요구하는지, 성능은 어떤지 등을 확인해보려고 간단하게 벤치마크를 구성해서 돌려봤다.

예상되는 초기 데이터 크기는 1000만개 정도.
벡터의 크기는 256, dot product를 사용할 예정이었다.
처리 시간은 빠르면 빠를수록 좋지만, 밀리초 단위만 보장되더라도 충분했다.

시스템 리소스도 적으면 적을수록 좋았다. 확실한 유용성이 아직 검증되지 않았으므로, 일단 가볍게 시작해서 데이터셋과 사용량이 증가한 다음에 본격적으로 옮기는 것도 검토해볼만했다.

그래서 단일 노드에, 메모리는 8GB, CPU는 단일코어로 가정하고 얼마나 잘 버티는지를 검증했다.

후보는 PostgreSQL(pgvector), Elasticsearch 정도다.
milvus 같은 전문 vectordb도 생각은 해봤지만, 충분한 사용량이 생긴 뒤에 옮겨도 무방하다고 판단했다.

테스트에 사용한 전체 소스코드는 깃헙에 있다.
https://github.com/myyrakle/benchmarks/tree/master/vector_search




벡터 생성

먼저 테스트에 사용할 벡터열 1000만개를 사전에 생성했다.

use rand::Rng;
use std::error::Error;
use std::fs::OpenOptions;
use std::io::{BufWriter, Write};
use std::time::Instant;

const NUM_VECTORS: usize = 10_000_000;
const VECTOR_DIM: usize = 256;
const OUTPUT_FILE: &str = "vectors.txt";

fn main() -> Result<(), Box<dyn Error>> {
    println!(
        "Generating {} vectors of dimension {}...",
        NUM_VECTORS, VECTOR_DIM
    );
    let start_time = Instant::now();

    let file = OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(OUTPUT_FILE)
        .unwrap();
    let mut writer = BufWriter::new(file);

    let mut randomizer = rand::rng();

    for i in 0..NUM_VECTORS {
        let vector: Vec<f32> = (0..VECTOR_DIM)
            .map(|_| randomizer.random_range(-1.0..1.0))
            .collect();

        let magnitude: f32 = vector.iter().map(|&val| val * val).sum::<f32>().sqrt();

        // 각 요소를 크기로 나누어 정규화
        let normalized_vector: Vec<half::f16> = vector
            .iter()
            .map(|&val| half::f16::from_f32(val / magnitude))
            .collect();

        // 벡터를 문자열 레코드로 변환
        let record: Vec<String> = normalized_vector
            .iter()
            .map(|&val| val.to_string())
            .collect();

        // 진행 상황 표시 (선택 사항, 성능에 약간 영향 줄 수 있음)
        if (i + 1) % 1_000_000 == 0 {
            println!("Generated {} vectors...", i + 1);
        }

        let row_string = record.join(",");

        // 파일에 벡터 문자열을 append로 저장
        writer.write_all(row_string.as_bytes())?;
        writer.write_all(b"\n")?;
    }

    let duration = start_time.elapsed();
    println!(
        "Successfully generated and saved {} vectors to {}",
        NUM_VECTORS, OUTPUT_FILE
    );
    println!("Total time taken: {:?}", duration);

    Ok(())
}

뽑으니까 7기가 정도의 텍스트가 나왔다.

데이터베이스는 pgvector(hnsw용), pgvector(ivfflat용), elasticsearch 정도만 고려했다.

도커 컨테이너 기반으로 테스트 환경을 구성했다.

데이터 초기화

인덱스를 전부 구성해놓은 상태에서 만들어뒀던 벡터 1000만개를 밀어넣었다.

단순성을 위해 순진하게 하나씩 순서대로 넣는 방식을 취했다.

그렇게 했을때

pgvector(IVFFLAT)는 9시간 정도 걸렸고

pgvector(hnsw)는 특히나 오래 걸렸다. 메모리 병목에 걸린건지 뭔지는 잘 모르겠다.

확실한건 전반적인 리소스 소모량이 압도적으로 많았다는 것이다. 디스크 I/O도 무슨 테라급으로 소모했다.


pgvector(hnsw)는 3일 정도 걸렸다.


elasticsearch는 25시간이 걸렸다.


평상시 리소스 사용량은 대략 이렇다.
ivfflat이 유독 리소스를 적게 먹는 편이긴 하더라.


qdrant는 삽입 속도는 준수하고 메모리 사용량도 낮았는데, 가끔 CPU가 훅 치고 yellow가 뜨는 구간이 있었다. 뭔가 모아서 한번에 디스크에 압축하거나 하는 패턴이 있는 것 같다.




다만

실제 응용 환경에서는 초기화에 걸리는 시간이 이와는 크게 다를 것이다.

일단 벡터 임베딩을 모델로부터 받아오는 딜레이가 존재할 것이다. 이건 느려질 수 있는 요인이다.

보통의 DB 환경에서는 이보다 리소스가 넉넉할 것이다. 나는 1cpu로 빡세게 걸어놔서 스로틀링이 과하게 걸렸을 것이다. 보통 메모리를 8 잡으면 코어는 2~4개쯤 넣는다.

그리고 데이터 추가시에도 batch 단위로 모아서 insert를 하면 처리시간을 좀 높일 수 있다.

(pgvector의 경우) 인덱스를 만들기 전에 벡터를 밀어넣고, 데이터가 있는 상태에서 인덱스를 한번에 생성할 수도 있다. 워커 메모리 충분히 주고 돌리면 이보다는 빨리 될 것이다.


자, 그럼 준비가 됐으니 벤치를 적당히 찔러보겠다.




디스크 사용량

디스크 사용량은 DB별로 차이가 좀 나긴 했다.

elasticsearch가 가장 많은 공간을 차지했고


qdrant가 그래도 벡터 전문이라고 가장 컴팩트하게 썼다.




pgvector(IVFFLAT) 벤치마크

로직에 별로 특별한 부분은 없다.

쿼리는 100번 정도만 찔러보게 했다.
그리고 동시처리 태스크느 16개 정도만 잡았다.


가져올 데이터는 10개로 잡았다.


최대 레이턴시가 300ms 정도까지 나와서엄청 빠르다고 할 정도는 아니었지만, 평균 처리속도는 크게 나쁘지 않았다.

DB측의 CPU 리소스 압박은 약간 있었다.




pgvector(HNSW) 벤치마크

hnsw는 꽤나 빠르고 안정적이었다.

확실히 레이턴시가 lvfflat에 비해서 짧고 안정적이었다.




elasticsearch 벤치마크

elasticsearch도 비슷하게 구성해서 벤치를 때렸다.

생각했던 것 보다는 레이턴시가 저조했다.

샤드를 나누고, 정확도를 낮추고, 성능을 높이게끔 옵션을 조정하면 성능이 더 나오겠지만, 일단은 여기서 기본적인 단일노드 테스트로 마무리해보려고 한다.




qdrant 벤치마크

qdrant는 전용 VectorDB인 것에 비해 평균 처리량은 생각보다 그리 빠르지는 않았다.

3자리 밀리초 정도로 처리가 되었다.


그리고 돌때 CPU 리소스를 조금 먹긴 했다.

이것도 효율적으로 쓰려면 옵션 조정이 필요할 것 같다.




정리

특이하게도 레이턴시만 보자면 pgvector-hnsw가 꽤나 빨랐다. elasticsearch와 qdrant 같은 경우에는 뭔가 리소스 요구량이 더 많아서인지 모르겠는데, 이 정도 각박한 사양에서는 아주 그렇게 빠르지는 못했다.

다만 qdrant가 레이턴시보다는 메모리 사용량에 있어서 장점이 두드러진 것 같다. 1000만개 넣었는데도 평상시 메모리 점유율이 1기가 조금 넘는 수준이더라. 다른 DB들은 평상시에도 메모리의 대부분을 썼는데, 리소스 효율성 자체는 압도적이었다.

추가 DB 구성을 반기지 않는다면 pgvector-ivfflat도 꽤나 쓸만한거같다. 데이터 규모가 커질때 옵션을 조정하는게 까다롭고, 레이턴시가 조금 튀다는 단점은 있다.

초기버전에서 "사전 필터링" 같은 고급기능이 필요하지 않다면 pgvector도 고려해볼만한 것 같다.
기존에 검색엔진으로 elasticsearch를 쓰던 상황이면 elasticsearch 또한 고려해봄직하다.

하지만 둘다 쓰지 않았거나, 정말 수천만을 넘어가는 데이터셋을 적은 리소스로 효율적으로 관리하고 싶다면 qdrant가 좋은 선택일 것 같긴 하다.
