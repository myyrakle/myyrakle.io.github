#LLM
[원본 링크]

MCP는 앤트로픽에서 공개한, LLM을 위한 통신 프로토콜이다.
Model Context Protocol의 축약이다.



기본 원리

요즘 엄청난 혁신인 것처럼 띄워대서 착각할 수도 있는데, 이것 자체가 대단한 동작을 하는 것은 아니다.
LLM이 주체적으로 데이터를 요청하거나 가져올 수 있는 표준적인 인터페이스로 기능할 뿐이다.

결국 누구나 만들 수 있고, 누군가 하긴 했어야 하는건데 앤트로픽이 총대매고 시작하니까 우르르 몰려든 것이다.

예를 들어, 데이터베이스의 데이터를 LLM이 가져가서 참고할 수 있게 하고 싶다면, 데이터베이스를 조회하는 MCP 서버를 만들고, LLM이 그걸 찔러서 쓰게 할 수 있게 해주는 식으로 응용이 된다.

MCP 클라이언트는 IDE나 기타 도구들에서 구현해서 제공해주기 때문에, 뭔가 데이터소스를 추가하고 싶다면 MCP 서버를 직접 구현해서 쓰거나 구현체를 가져다가 실행시켜서 쓰면 된다.




레이어

프로토콜이라 하니 뭔가 거창해보이는데, 사실 여기서 직접 구현한건 별로 없다. JSON-RPC 위에 인터페이스만 끼워맞춘 것이다.
정확히 말하면 HTTP 위에 JSON-RPC를 얹고, JSON-RPC 위에 MCP를 올린 형태다.
그래서 HTTP SSE를 적극 활용하는 부분도 있다.




리소스 단위

MCP는 3가지 리소스 단위를 제공한다.
그것이 Tool, Resource, Prompt다.

1. Tool
일반적으로 사람들이 열광하는 "데이터 소스 연동"이 Tool이 제공하는 것이다.
데이터베이스에서 데이터를 긁어오거나, 피그마에서 디자인 데이터를 받아오거나, 실시간 날짜 정보를 받아오거나 하는 것들을 이걸 통해서 구현한다.

2. Resource
Resource의 경우에는 채팅에 사용할 일종의 문서 데이터들을 보관하는 저장소라고 할 수 있다.
예를 들면 코드 작성 가이드라인을 마크다운 파일로 만들어놓고 LLM이 사용하게 하고 싶다면, 이런게 Resource를 사용하기에 적절한 사용례다.

3. Prompt
Prompt는 프롬프트 템플릿들을 저장해놨다가 필요할때 꺼내쓰는 단순 저장소 개념이다.



서버를 구성하거나 클라이언트로 연결하는 건 다음 포스트에 다뤄보겠다.
