[Go] Gorm - ERROR: cached plan must not change result type (SQLSTATE 0A000)

  • 2024.10.16
  • ::
  • 1 min read
#Go
[원본 링크]

다른 환경은 모르겠는데, gorm-postgres에는 치명적인 기능 결함이 있다.
테이블에 변경사항이 생기면, 하위호환이 보장되는 변경이라도 기존 기능이 뻗는 문제다.


package main

import (
	"gorm.io/driver/postgres"
	"gorm.io/gorm"
)

type User struct {
	ID   int
	Name string
}

func main() {
	dsn := "host=localhost user=postgres password=... dbname=postgres port=5432 sslmode=disable"
	db, err := gorm.Open(postgres.Open(dsn), &gorm.Config{})
	if err != nil {
		panic(err)
	}

	if err := db.AutoMigrate(&User{}); err != nil {
		panic(err)
	}

	// 스레드 5개 띄워서 무한 루프 돌리기
	for i := 0; i < 5; i++ {
		go func() {
			for {
				var user User
				if result := db.Find(&user); result.Error != nil {
					panic(result.Error)
				}
			}
		}()
	}

	for {
	}
}

그냥 이렇게 꽂는거 돌려놓고


컬럼 하나만 추가해도


뻗는다.

이슈 들어가보니 postgres 드라이버 핑계를 대는데, entgo 같은건 이런 문제가 없다.




PreferSimpleProtocol 사용

postgres config를 명시적으로 넣고 PreferSimpleProtocol라는 특수 옵션을 넣으면 저 캐시 문제를 회피할 수 있다. 구현을 이상하게 해두고 왜 옵션을 켜서 해결하라는건지 이해는 잘 안되지만...

아무튼 이러면 최소한 캐시 문제로 터지지는 않는다.

근데 이게 prepared statement 캐싱을 비활성화하는거라고 하는데, 내가 예전에 시도해봤던 기억으로는 캐싱만 비활성화하는게 아니라 드라이버의 전반적인 동작 방식이 바뀐다.

기존 동작이 전부 호환되지 않고 타입 변환같은게 개떡같이 바뀌어서 시도하다가 그냥 포기했던 기억이 있다.
도대체 어떻게 만든거지?

그냥 Gorm을 쓰지 않는걸 권한다.
