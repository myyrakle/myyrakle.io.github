[Rust] 최적화: Bound Check

  • 2024.09.04
  • ::
  • 3 min read
#Rust
[원본 링크]

Bound Check는 Rust에서 위험한 접근을 막기 위해 기본적으로 제공하는 방어 수단 중 하나다.

Rust에서 Array나 Vector 같은 배열 리소스에 접근할 때는 접근하려는 인덱스가 유효한 인덱스인지를 검사하는 코드가 자동으로 삽입된다.
만약 유효한 인덱스가 아니라면 out of range panic을 발생시킨다.
이걸 bound check라고 하는 것이다.

이를 통해서 Rust는 C/C++ 기반 코드에서 밥먹듯이 발생하는 버퍼 오버플로 같은 문제를 최소화할 수 있다.




성능 문제?

Bound Check 자체도 추가적인 분기가 들어가는 것이다보니 비용이 완전히 없는 것은 아니다.

근데 그렇다고 해서 성능에 큰 무리를 줄 정도도 되지 못한다.
Bound Check를 제거하더라도 보통은 1~3% 정도의 미미한 성능 차이가 발생할 뿐이고, 극적으로 효과가 나타나는 경우라도 10% 정도의 최적화가 한계다.

그래서 일반적인 응용프로그램에서는 고려할 필요가 거의 없다. 그걸 알아두길 바란다.




예제코드

한번 간단한 예제코드 돌려보면서, 실제로 성능차이가 어떻게 나는지를 관측이라도 해보자.

#[inline(never)]
fn add(sum: &mut i128, nums: &Vec<i32>, i: usize) {
    *sum += nums[i] as i128;
}

fn main() {
    let mut nums = vec![];

    for i in 0..1000000000 {
        nums.push(i);
    }

    let mut sum: i128 = 0;

    let mut i = 0;

    let start_time = std::time::Instant::now();

    while i < nums.len() {
        add(&mut sum, &nums, i);

        i += 1;
    }

    let elapsed_time = start_time.elapsed();

    println!("Sum: {}", sum);

    println!("Elapsed time: {:?}", elapsed_time);
}

일부러 무지막지한 규모로 돌도록 했다.


1초 조금 좀 넘게 걸렸다.


어셈블리를 까보면, 이런 식으로 panic_bounds_check라는 함수를 호출하는걸 볼 수 있다.
이게 bound check를 처리하는 녀석이다.

release 빌드에서는 최적화때문에 다 날라가지만, debug 빌드에서는 이런 core 함수를 호출해서 배열 접근을 수행한다.

여기서 값을 가져오는 동시에 bound check까지 해주는 것이다.




Bound Check 제거하기: unsafe

Bound Check를 제거해서 성능을 높이는 가장 단순한 방법은, bound check를 하지 않는 unsafe 함수를 사용하는 것이다.

그냥 교체만 하면 된다. 대신 그 위험은 스스로 책임져야 한다.

그럼 bound check가 깔끔하게 도려내진다.


하지만... 일반적인 상황에서는 성능차가 의미있게 드러나지는 않는다. 캐시 때문일 수도 있다.
#[inline(never)]를 제거해서 인라이닝을 시키면 unsafe 버전이 조금 더 빨라질 때도 있는데, 그것도 의미있는 격차가 벌어지지는 않더라.




Bound Check 회피하기: safe

unsafe부터 박기보다는, 간접적인 방법을 통해 회피하는 것을 권하고 싶다.
다행스럽게도 약간의 트릭을 응용하면 Bound Check가 비활성화되도록 유도할 수 있다.

예를 들어, 이런 코드가 있다고 가정하겠다.

#[inline(never)]
fn do_something(input: &[i32; 3]) -> i32 {
    let a = input[0];
    let b = input[1];
    let c = input[2];

    a + b + c
}

fn main() {
    let mut nums = vec![];

    for i in 0..1000000 {
        nums.push(i);
    }

    let result = do_something(&nums);

    println!("Result: {}", result);
}

이 코드는 당연히 bound check를 수반하는 코드를 생성한다.
저기서 뻘겋게 bound check 호출하는게 그거다.

이러한 패턴은 놀랍게도, assert을 응용해서 최적화를 유도할 수 있다.

이런식으로 배열 접근 전에 무조건 3보다 크다는 것을 보장해주면


bound check가 다 날라간다.
바로 앞에서 미리 체크해서 패닉을 던지기 때문에, Bound Check가 필요없다는 것을 컴파일러가 깨닫기 때문이다.

다만 debug_assert를 사용하면 debug 모드에서만 assert가 들어가기 때문에 release에서는 최적화가 되지 않는다는 점을 유의한다.




iterator 활용하기

Rust가 자랑하는 캐치프라이즈 중 하나다.
iterator 기반의 루프를 활용하면, Bound Check가 거의 생성되지 않는다. 애초에 iterator 자체가 정해진 범위에 대한 반복을 가정하는 구조를 갖기 때문이다.

Bound check도 없고, 최적화도 거의 일반 루프와 대등한 수준으로 나온다.
