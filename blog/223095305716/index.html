#Terraform
[원본 링크]

테라폼을 CLI를 통해 사용해보면, 기본적으로는 민감한 변수값들이나 배포 상태에 대한 값(tfstate)들은 로컬에 저장을 하게 된다.

이걸 지속적으로 관리하려면 그렇게 생겨난 부산물들을 전부 어디다 넣고 관리를 해줘야 한다. 여기에는 git 같은걸 써도 되고 그렇게 많이들 쓰기도 하지만, Terraform Cloud이 그에 대한 가장 표준적이고 권장되는 해결책이다.




비용

무료 사용량이 널널한 편이다.
최대 5인 팀까지는 무료고, 그 이상은 요금제를 써야한다.
근데 배포 팀 인원이 5명이 넘는 경우가 얼마나 있겠나?

중소규모에서는 그냥 도입해도 무방할 것 같다.





조직 만들기

조직을 먼저 만들어야 한다.
조직 이름은 전세계에 고유하니, 겹치지 않게 잘 짓는다.

그래서 이런 대시보드에 진입하면 잘 만들어진 것이다.




변수 집합(variable set)

이건 조직 내에서 모든 작업영역들이 공유할 수 있는 값들을 정의할 때 쓴다.

이를테면, AWS 리소스에 접근하기 위한 AWS 인증 값 같은게 있겠다.
여러 작업단위들이 다 AWS로 관리된다면 이건 변수 세트로 빼서 관리하는게 용이하다.

센서티브 옵션 켜서 노출위험을 줄이고 환경변수로 집어넣는다.

이렇게.




워크스페이스 만들기

워크스페이스는 하나의 배포단위라고 보면 된다.
서버를 하나 띄우고 싶다면 그걸 테라폼으로 말아서 워크스페이스로 띄우면 된다.

테라폼 코드를 어떻게 받아올지를 먼저 정한다.

나는 version control workflow로 했다.


그리고 소스코드 공급자를 연결하고, 레포지토리도 골라준다.
나는 github이었고, 이걸 썼다.
https://github.com/myyrakle/terraform


적당히 연결하고, 작업 이름 짓고


만들어준다.




배포하기

우선 연결한 github 레포지토리 루트경로에 테라폼 환경이 구성되어있어야 한다.
내가 사용한 레포지토리는 이랬다.
https://github.com/myyrakle/terraform

변수 2개가 있

// 리전
variable "region" {
  description = "region"
  type        = string
}

// 서버명
variable "server_name" {
  description = "server name"
  type        = string
}

ec2 하나만 덜렁 띄우는 단순한 코드다.

terraform {
  required_providers {
    # 일종의 라이브러리 로드
    aws = {
      source  = "hashicorp/aws"
      version = "~> 4.16"
    }
  }

  required_version = ">= 1.2.0"
}

provider "aws" {
  region = var.region
}

resource "aws_instance" "server" {
  tags = {
    Name = var.server_name
  }

  ami           = "ami-0e9bfdb247cc8de84"
  instance_type = "t2.micro"
}
