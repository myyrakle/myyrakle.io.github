[Quarkus] 소개

  • 2026.03.11
  • ::
  • 3 min read
#Java
[원본 링크]

Quarkus는 최근에 나온 자바 기반의 서버 프레임워크다. 레드햇에서 만들었고, 완전한 오픈소스다.
2019년에 시작되었으며, 아파치 라이센스로 관리된다.

무겁고 느린 Java Spring 대신, 컨테이너에 서버를 빠르게 띄우기 위한 목적에서 나왔다고 보면 된다.
빠른 로딩을 위해 graalvm 기반으로 컴파일된다.



vs Spring

솔직히 말해서 Spring이 하지 못하는 것을 Quarkus가 더 잘한다고 말할 수는 없다.
이건 취향의 문제도 있고, 선택의 여지가 좀 크다.

Quarkus의 장점으로 꼽을 수 있는 것은, 일반적으로 더 적은 코드로 더 간결하게 작성할 수 있고, 문서화 수준이 높으며, 릴리즈도 제법 활발하게 이뤄진다는 것이다.
게다가 GraalVM과의 통합이 잘 되어있고, 컨테이너 환경을 고려한 주변 도구들이 제법 잘 갖춰져있다.

일반적인 생태계의 크기는 현격한 차이가 존재한다. Spring의 모듈들이 반드시 뛰어다는 것은 아니지만.




설치

먼저 전용 CLI부터 설치해야 한다. 유닉스 계열이면 curl 명령을 통해 설치할 수 있다.

curl -Ls https://sh.jbang.dev | bash -s - trust add https://repo1.maven.org/maven2/io/quarkus/quarkus-cli/
curl -Ls https://sh.jbang.dev | bash -s - app install --fresh --force quarkus@quarkusio

설치하고 나면 quarkus라는 프로그램이 실행 가능해져야 한다.




Hello World 시작해보기

방금 설치한 quarkus CLI로 프로젝트를 부트스트래핑해보겠다.
다음과 같이 프로젝트명과 프로젝트 디렉터리명을 지정해서 create app 명령을 날리면 된다.

보통 다른 도구들은 프로젝트명 지정하면 그거랑 동일하게 디렉터리명도 생성해주는데, 이건 아니더라.
자바 아니랄까봐 UX가 은근히 좀 멍청하고 구리다.

quarkus create app --name quarkus_start -o quarkus_start

아무튼 만들면 이런 식으로 프로젝트가 초기화되어있을 것이다.

기본값은 maven이다.
코드 형태는 대략 스프링과 비스무리하긴 하다.

그러면 CLI의 dev 명령을 통해 즉시 dev 서버를 빌드해볼 수 있다.

뭐 잔뜩 다운받고 실행을 하게 된다. 패키지를 잔뜩 받아서 최초 부팅은 좀 느리다.

아무튼 실행이 된다면 8080 포트를 통해 정의된 API를 즉시 사용해볼 수 있다.

그리고 dev 모드로 실행한 경우에는 루트 경로로 진입시에 dev UI가 뜬다.

별로 대단한 것은 아니고, 어떤 API가 있고 어떤 것들로 구성되어있는지를 간단히 보여주는 대시보드다.


디펜던시 그래프도 있는데, 이건 굳이 왜 넣은건지 모르겠다. 쓰는 사람이 있나?




컨테이너 배포

생성된 프로젝트를 까보면 dockerfile들이 몇개 있다. 이거 그대로 말아서 쓰면 된다.

# JVM 버전으로 실행
./mvnw package -DskipTests
docker build -f src/main/docker/Dockerfile.jvm -t quarkus-jvm:latest .
docker run --name quarkus-jvm -p 8080:8080 --rm quarkus-jvm:latest

# GraalVM 버전으로 실행
./mvnw package -Dnative
docker build -f src/main/docker/Dockerfile.native -t quarkus-graal:latest .
docker run --name quarkus-graal -p 8081:8080 --rm quarkus-graal:latest

# GraalVM 경량 버전으로 실행
./mvnw package -Dnative
docker build -f src/main/docker/Dockerfile.native-micro -t quarkus-micro:latest .
docker run --name quarkus-micro -p 8082:8080 --rm quarkus-micro:latest

근데 또 귀찮게, docker build만으로 되는게 아니라 사전에 명령 하나 써서 target에 중간 결과물을 둬야 한다. 번잡스럽다.

아무튼 적당히 빌드하면, 이미지를 그대로 가져다가 실행해서 쓰면 된다.

그러면 각 버전별로 리소스 차이가 어떻게 날까?

docker image의 경우에는 크기 차이가 제법 난다.
딱 기본적인 hello world 코드만 있는 것인데도 JVM 버전은 400MB가 넘고, graal 버전은 150MB 정도다. graal 버전이 최소 2.5배 정도는 가벼운 것이다.


기본 메모리 사용량에도 차이가 있다.
JVM 버전은 200MB 가까이 먹지만, graal 버전들은 10MB도 먹지 않는다. 확실히 가볍긴 가볍다.
