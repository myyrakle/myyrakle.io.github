[C++] 람다(lambda) 함수

  • 2018.02.10
  • ::
  • 3 min read
#C++
[원본 링크]

람다는 익명함수나 무명함수라고도 부르는 기능으로, C++11부터 표준에 추가되었고 C++14에서 한 차례의 개선이 이루어졌다.

람다함수의 역할을 짧고 어렵게 말하면 '클로저 객체를 생성'하는 것이라고 할 수 있다.

근데 뭐 사실 어려울건 없고 그냥 함수 스코프 내에서 함수를 정의하고 그 함수의 포인터를 얻어낼 수 있는 것이라고 보면 된다.

대략적인 형태는 다음과 같다.

[ capture ] { expression } //파라미터 없을경우
or
[ capture ] ( parameter ) { expression }
or
[ capture ] ( parameter ) ->return-type { expressiom }

캡쳐는 일단 뒤로 미루고,

파라미터는 통상 함수의 파라미터와 완전히 동일하다.

C++14부터는 매개변수에 auto를 쓸수있다.(제네릭 람다. 함수템플릿과 유사하다.)

함수 몸통도 통상 함수와 큰 차이는 없다. 그런데 C++11에서는 몸통 안에서의 지역변수 선언이 불가하지만 C++14에서는 허용된다.

반환타입은 써줘도 되고, 안써줘도 된다. 디폴트로 둘 경우 컴파일러가 알아서 추론해준다.

캡쳐는 외부에 존재하는 값을 가져다 쓸수 있는 기능이다. 갈무리나 획득이라고도 부른다.

람다함수 외부에 있는 var에 대해,
[var], [var,var2]... 이런식으로 하면 값 전달이라고 해서 람다 내에서도 변경이 불가하다.
[&var] 이건 참조 접근이다. 람다 내에서 값 변경이 가능하다.
[=] 외부에 있는 모든 값을 값으로 접근한다.
[&] 모든 값을 참조로 접근한다.


그리고 C++14부터는 초기화 캡쳐라고
[num=5+7] 이런식으로 람다 내에서 쓸 변수를 선언할 수 있다.
그냥 변수명만 붙이는 일반 캡쳐와 다르게 거의 모든 표현식을 다 넣을 수가 있다.

무브시맨틱도 가능하다. 이걸 응용해서 이동클로져를 구현할 수 있다.
[hmm = std::move(some)](){...};



람다도 일반함수처럼 재귀호출이 가능하다. 근데 굳이...?


자 그럼 이제 람다를 써보자.

int main()...
[] (int n) { std::cout<<n<<std::endl; };
...

이러면 람다함수가 생성된다.
근데 이러고 바로 컴파일하면 아무것도 출력되지가 않는다.
그냥 함수만 만들어놓고 아무것도 안했기때문에 람다의 주소값이 생성됐다가 붕 떠버리는 것이다.

[] (int n) { std::cout<<n<<std::endl; } (10);
이러면 10이 출력될 것이다. 람다도 함수이기 때문에 바로 값 10을 전달해서 호출하는 것이다.

그리고 함수포인터에 저장해서 쓰는 방법이 있다.

auto Func = [] (int n)
{ std::cout<<n<<std::endl; };

함수포인터나 펑셔널헤더의 std::function에 저장할 수도 있겠지만, 마이어스 형님의 글에 따르면 function은 auto로 추론해서 쓰는 것보다 성능이 후달리고 타이핑하기도 빡세기 때문에 안쓰는게 낫다고 한다.

어쨌든 저러면 이렇게
Func(10);
함수와 같이 사용할 수가 있다.

근데 이것만으로는 이런 기능이 굳이 필요한가? 하는 의심이 들 수 있을 것이다.

람다의 진가는 함수 객체 펑터(Functor)가 사용될 때 드러난다.

algorithm 헤더의 for_each 함수를 이용해서 벡터 컨테이너의 멤버를 전부 출력하는 코드를 짠다고 치자.
for_each는 시작 위치 이터레이터와 끝 위치 이터레이터를 받아서 범위를 지정하고, 함수 객체를 마지막 인자로 받아서 개개 멤버를 함수객체의 방식대로 접근하게 해주는 함수다.

class Functor
{...};

int main()
{
    ......
    vector v_arr{1,2,3,4,5};
[for_each(v_arr.begin](http://for_each(v_arr.begin)(), v_arr.end(), Functor() );
}

람다가 등장하기 전에는 다 이런식으로 썼었다.

for_each의 세번째 인자로 함수의 포인터를 넣어도 되지만 함수객체가 성능이 더 좋다고 하더라.

자, 이 코드의 문제점이 무엇일까

분명 정상적으로 작동하고 딱히 문제되는 건 없어보인다.
가독성에도 문제가 딱히 없다.

그런데, 저 함수객체의 정의부와 함수객체 호출부 사이에 한 수십수백줄의 코드가 박혀있다고 생각해보자, 아니면 함수객체가 아예 다른 파일에 분할작성되었을 수도 있다.

그러면 해당 함수객체가 호출되는 부분을 보고 이해하거나 고치려면 그때마다 함수객체 정의부로 이동하거나 피킹해서 확인을 해야한다.
별거 아닌것같지만 이런게 쌓이고 쌓이다보면 큰 불편요소가 되고 가독성이 나빠진다고 할 수 있다.

하지만 람다를 사용하면 이러한 문제점을 제거할 수 있다.

[for_each(v_arr.begin](http://for_each(v_arr.begin)(), v_arr.end(), [] (int n) { std::cout<<n<<std::endl; } );

스크롤 올리거나 피킹할 거 없이, 바로 눈에 딱 들어온다.

이게 바로 람다다.
