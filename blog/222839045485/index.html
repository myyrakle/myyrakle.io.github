이번 포스트에서는 git merge에 대해서 다뤄보겠다.

merge는 말 그대로 병합을 하는 기능이다.
브랜치끼리 합쳐준다.




기본 merge

그래서 이게 왜 필요한걸까?

git으로 작업을 하다보면, 기능별로 나누기 위해서든, 분업을 하기 위해서든, 배포를 관리하기 위해서든, 아무튼 브랜치를 쪼개서 개발을 하게 된다.

만약 master를 기본 배포 브랜치로 하고 develop을 개발용 브랜치로 한다면,
develop으로 먼저 작업을 해서 어느정도 작업이 완성되고 난 다음에, develop의 작업사항을 master에 통합해야 하는 식으로 진행이 된다.

그럼 이런식으로 develop에 commit을 하다가

이렇게 merge로 master에 합쳐주면 되는 것이다.

그런데 이렇게만 되면 참 좋겠지만... 현실은 그렇지 않다.




merge conflict

위의 경우에는 깔끔하게 맞물려있는 구조라서 Fast-forward로 바로 직통 merge가 성공했는데, 문제는... 그렇지 않을 경우가 많다.

만약 병합하는 2개의 브랜치가 하나의 소스파일에 대해서 각각 수정을 했다면, 어느놈을 반영해줘야 하겠나?
이게 자연스럽게 겹쳐진다면 git이 알아서 해결을 해주기도 하는데, 그렇지 않다면 충돌을 직접 해결해야 한다.

한번 상황을 재현해보겠다.

master에서는 저렇게 수정을 하고

develop에서는 이렇게 수정을 했다.

이 상황에서 merge를 한다면, 무조건 충돌이 발생한다.

그도 그럴것이, lhs_1로 바꾸는게 맞는지, lhs_2로 바꾸는게 맞는지 git이 어떻게 알겠나?

그래서 이런식으로 선택지를 준다.

Accept Current Change를 선택하면 현재 브랜드로 덮어씌우고, Accept Incoming Change를 하면 가져오는 브랜치로 덮어쓴다.
Accept Both Change로 하면 둘다 합쳐버리긴 하는데... 억지로 합쳐버리는지라 결과가 이상할때가 더 많아서 쓰지 않는걸 권한다.

뭘로든 선택을 하고 코드를 정리했다면, 커밋찍고 마저 작업하면 된다.




squash merge

squash는 merge 사항을 하나로 뭉쳐서 넣어버리고 싶을때 사용할 수 있다.

예를 들어, 하나의 merge 작업사항에 여러개의 커밋이 들어있다면

당장 작업할때는 괜찮지만, merge할때 대량으로 커밋내역이 우르르 쏟아져서 보기가 좀 불편할 수가 있다.

그럴때 사용하는 기능이다.

이런식으로 커밋이 여러개 뭉쳐있는 상황에서 merge를 할때

--squash 옵션을 주고

커밋메세지를 새로 써주면

커밋을 하나로 뭉쳐서 만들어준다.
근데 이건 master에만 뭉쳐준거고, develop에는 하나하나 다 남아있다.




rebase merge

이건 전용 명령어가 있는건 아니고, rebase와 merge를 연달아 사용해서 처리하는 idiom을 말한다.

이 merge를 사용하면, merge할 커밋들을 대상 브랜치에 억지로 쑤셔넣는다.
뭉치지도 않고 그대로, 브랜치와의 연관성 없이 말이다.

develop에 커밋을 먼저 몇개 세팅했다.

방법은 간단하다.
merge하는 것과 반대로 rebase를 하고, 다시 반대로 돌아와서 merge를 하면 된다.

rebase할때 conflict가 좀 날수도 있는데, 그건 알아서 잘 해결하자.

그러면 이런식으로 메세지가 그대로 다 들어가있을 것이다.

근데 이걸 쓸일이 있나?
잘 모르겠다.




Pull Request (Github)

Github pull request에서 merge를 할때도 종류를 선택할 수 있다.
동작은 위에서 설명한 것들과 동일하다.
