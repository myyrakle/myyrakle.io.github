[PostgreSQL] Lock

  • 2022.10.26
  • ::
  • 3 min read
#PostgreSQL
[원본 링크]

lock은 동시성 제어에서, 동시에 어떤 항목에 접근을 시도할때 어떻게 충돌을 방지하거나 허용할 것인지에 대한 기능이다.



테이블 Lock 걸기

이게 왜 필요하고, 어떨때 사용하는지부터 사용사례를 한번 간략하게 보겠다.

만약 데이터 풀을 관리하는 테이블이 하나 있고,

어떤 서버에서 이걸 순서대로 하나씩 꺼내쓴다고 생각해보자.
꺼내쓰고 나서는 used에 true값을 새겨주고, 한번에 여러 요청에서 동일한 값을 가져와서는 안 된다.

아무 생각없이 이런식으로 하나씩 꺼내쓰는 것은 문제의 소지가 많다.

무작정 used가 false인 것들을 가져오다보니, 여러개의 요청에서 동시에 동일한 pool 데이터를 읽고, 동일한 pool 데이터에 used true를 새길 수가 있는 것이다.

그래서 저러면 그냥..

9개가 데이터 행 하나만 쓰고 만다.

이런 경우에 필요한 것이 Lock이다.
table lock을 사용하면 현재 트랜잭션이 종료되기 전까지는 다른 트랜잭션에서 접근을 할 수가 없다.

lock table 테이블명;

항상 트랜백션 구문 안에서 사용되어야 한다는점 있지 말자.

이렇게 했다면 이제

순차적으로 잘 접근이 되었을 것이다.




행 단위 lock

그런데 사실, 저 행 하나씩 써먹는데 한 쿼리 돌때마다 테이블 전체를 lock해버리는건 다소 낭비일 수도 있다.
사실, 같은 행에 동시접근만 하지 않고, 다른 행은 접근해도 되는게 아닌가?

그래서 이런 케이스에는 행 단위 lock을 거는 편이 보다 자연스럽다.
대략적인 사용법은 이렇다.

... 쿼리 ... for update

이러면 저 조건에 의해 가져온 항목들은 현재 트랜잭션이 끝날 때까지 lock이 걸려서 접근하지 못한다.

이 케이스에서는 사실 처음 쿼리를 돌릴때는 전부 첫번째 행을 가리켜서 실제 동작은 다르지 않겠지만, 다른 외부시스템에서 다른 행들에 대해서는 접근을 할 수 있다는 것이 달라졌다.




Lock의 종류

락 수준에는 꽤나 다양한 옵션이 있다.

lock table 테이블명 [ IN lockmode MODE ];

이것도 꽤 종류가 많다.


1. ACCESS SHARE

가장 쉬운 수준의 락이다.
ACCESS EXCLUSIVE와만 충돌하고, 그게 아니라면 언제든 접근이 가능하다.
일반적으로 테이블을 읽기만 하고 테이블을 수정하지 않는 모든 쿼리는 이 락으로 동작한다.


2. ROW SHARE

EXCLUSIVE, ACCESS EXCLUSIVE 락과 충돌한다.
for update 같은 행 단위 lock을 걸 경우에 이게 붙는다.


3. ROW EXCLUSIVE

SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE와 충돌한다.
UPDATE, DELETE, INSERT 등의 수정 쿼리가 기본적으로 이걸로 동작한다.


4. SHARE UPDATE EXCLUSIVE

SHARE UPDATE EXCLUSIVE, SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE 락과 충돌한다.

VACUUM, ANALYZE, CREATE INDEX CONCURRENTLY, CREATE STATISTICS, COMMENT ON, REINDEX CONCURRENTLY, 일부 ALTER INDEX에 의해 획득된다.


5. SHARE

ROW EXCLUSIVE, SHARE UPDATE EXCLUSIVE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE와 충돌 한다.
이 모드는 동시 데이터 변경으로부터 테이블을 보호한다.

CREATE INDEX 등에 의해 동작한다.


6. SHARE ROW EXCLUSIVE

ROW EXCLUSIVE, SHARE UPDATE EXCLUSIVE, SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE와 충돌한다.

CREATE TRIGGER나 일부 ALTER TABLE 쿼리에 붙는다.


7. EXCLUSIVE

이게 락을 잡고 있는 경우에는 ACCESS SHARE lock을 들고있는 트랜잭션만 접근이 가능하고, 그 외엔 접근이 막힌다.


8. ACCESS EXCLUSIVE

가장 강한 수준의 lock으로, 모든 읽기든 쓰기든 간에 동시에 지금 돌고 있는 하나의 트랜잭션만 해당 테이블에 접속이 가능하다.
락 수준을 지정하지 않고 lock 구문을 사용한다면 이게 기본값으로 들어간다.
그리고 테이블 전체에 많은 영향을 주는 alter index나 alter table 쿼리의 기본 락 설정이 이거다.

좀더 이해를 돕기 쉽게 만들어놓은 표다.

어떤 lock이 어떤 lock에 접근이 가능한지를 표현해놨다.
