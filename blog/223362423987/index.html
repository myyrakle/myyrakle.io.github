이래나 저래나 Go는 GC 기반의 언어이고, 메모리의 할당과 해제가 성능상의 병목이 될 때도 분명히 존재한다.
그럴때 GC 부담을 줄이는 방법은 여러가지 접근법이 있지만, 가장 확실한건 아무래도 할당과 해제 자체를 자주 하지 않도록 유도하는 것이다.




내장 Memory Arena

그걸 위한 대표적인 기능 중 하나가 바로 memory arena라는 것이다.
Go 1.20부터 experimental 모듈로 추가되었다.

이건 작은 크기의 데이터 조각들을 부담없이 빠르게 할당하고 한번에 반환할 수 있도록 하는 것을 목적으로 한다.

사용 패턴 자체가 한번에 몰아서 확 쌓았다가 나중에 몰아서 정리해도 되는 경우에 적합하다.
그대신 일이 끝나면 수동으로 정리를 해줘야 한다. 그렇지 않으면 누수 문제가 생긴다.

사용법은 대략 이렇다.
arena라는 이름의 모듈로 import할 수 있고

실행할 때도 명시적으로 활성화를 해줘야 한다.

사용방법은 그다지 어렵지 않다.
먼저 전역 힙 공간을 생성해줘야 한다.

저게 일종의 짬통이 되는 셈이다.
사용이 다 끝나면 Free 함수로 할당한 객체들을 전부 날려준다.

그리고 객체를 할당할때는 New 함수를 통해서 수행할 수 있다.

각자 할당된 공간에 대한 포인터를 반환하고, 이를 통해 값을 수정하거나 조회할 수 있다.

그리고 위의 Free 함수가 실행되면 한번에 싸그리 정리되는 셈이다.
Free 함수가 실행된 이후에는 저 포인터 핸들들을 사용하면 안된다.

아래는 전체 예제 코드다.

package main

import (
	"arena"
	"fmt"
)

func main() {
	// arena 전역공간 생성
	mem := arena.NewArena()
	defer mem.Free()

	// 할당
	number := arena.New[int](mem)
	*number = 4444
	fmt.Println(*number)

	// 슬라이스 할당
	slice := arena.MakeSlice[string](mem, 100, 200)
	fmt.Println(len(slice), cap(slice))
	slice[0] = "Hello"
	slice[1] = "World"
	fmt.Println(slice[0], slice[1])
}

잘 동작한다.

다만 현재 arena 구현은 문제가 많아서 제거되거나 변경될 가능성이 많다고 한다. 상세한 내용은 이슈를 참고한다.
https://github.com/golang/go/issues/51317
적어도 근시일에 이 기능이 안정화될 일은 없는거같다.




대안: 직접 구현하기

범용성을 약간 포기한다면 아레나를 직접 구현하는 것도 어렵지는 않다.

할당해제를 막기 위해서는 root 객체가 그 객체의 포인터를 들고만 있어도 되는데, any 포인터든 뭘로든 데이터를 들고 있게끔 만들면 되기 때문이다.

혹은 모든 변수들을 하나의 구조체에 다 정의해놓고 상위 컨텍스트에서 생성한 다음에 포인터로 공유하는 식으로 사용해도 거의 동등한 동작을 기대할 수도 있을 것이다.
