마크앤스윕(Mark-and-Sweep)


배경 설명
new를 통해 동적으로 생성된 모든 객체는 힙 메모리에 할당이 됩니다.
이렇게 객체를 만들다보면 메모리 부족(Out Of Memory) 에러가 발생할 수도 있는데요, 그건 객체에 힙 메모리를 할당할수 없단겁니다.
그래서 새로운 객체들을 만들 공간을 확보하기 위해서라도, 프로그램에서 더이상 참조되지 않을, 혹은 도달될수 없는 객체의 메모리를 정리해줄 필요가 있습니다.
이 메모리는 수동으로 풀어줘야 하기 때문에 프로그래머에게 있어서 꽤나 골치아픈 녀석입니다.
하지만 걱정마세요. 여기에서 설명할 가비지컬렉션이 우릴 구원해줄 수 있습니다. 참조되지 않는 모든 객체에 대한 힙 메모리를 알아서 정리해주죠!!

백그라운드에서 수행되는 가비지컬렉션은 종류가 꽤나 다양한데요. 그중 하나가 바로 마크앤스윕입니다.


**마크앤 스윕 알고리즘 **
어떤 가비지컬렉션 알고리즘이든 전부 2가지의 기본 동작을 수행합니다.
첫번째, 도달할수 없는 모든 객체들을 탐지할수 있어야 한다.
두번째, 쓰레기 객체가 사용하는 힙 공간을 정리해서 다시 사용할수 있도록 해야한다.

그리고 위의 동작들은 마크앤스윕이라는 알고리즘으로 수행됩니다.
두가지 페이즈가 있죠

  1. 표시 페이즈(Mark phase)
  2. 쓸기 페이즈(Sweep phase)

Mark Phase
객체가 생성되면 비트를 0(false)로 표시합니다.
마크 페이즈에서는 모든 도달 가능한 객체. 또는 사용자가 참조할수 있는 객체에 1(true)가 찍힙니다.
이제 이 동작을 수행하려면 그래프 탐색(graph traversal)을 할 필요가 있습니다. 그리고 그 방법으로 깊이우선 탐색기법(depth first search approach)을 사용하죠.
여기에선 모든 객체를 노드처럼 취급하고, 노드에서 접근할 수 있는 모든 노드들을 방문합니다. 더이상 들를 곳이 없을때까지요.

루트는 객체를 참조하는 변수이며, 지역변수가 곧장 접근할 수 있습니다.

우리는 일단 루트가 하나만 있다고 간주할겁니다.

그리고 markedBit(obj) 의 호출로 객체의 mark bit에 접근할 수 있습니다.

알고리즘 -Mark phase:
Mark(root)
If markedBit(root) = false
then markedBit(root) = true
For each v referenced by root
Mark(v)

만약 루트가 하나 이상이라면 그냥 모든 루트에다 Mark를 호출하면 됩니다.


Sweep Phase
"sweep(쓸어버린다)"라고 이름을 붙였듯이, 이건 힙 메모리에 있는 모든 도달 불가능한 객체를 치워버립니다.
marked 값이 false로 설정된 모든 객체는 정리되고, true인 객체는 유지하죠.
그럼ㅎ이제 도달가능한 객체의 mark값을 false로 설정해서 다시 마크 페이즈로 넘어가 모든 도달가능한 객체에 마크를 하도록 해야합니다.


알고리즘 – Sweep Phase
Sweep()
For each object p in heap
If markedBit(p) = true
then markedBit(p) = false
else
heap.release(p)

마크앤스윕 알고리즘은 프로그램에서 직간접적으로 접근할 수있는 모든 객체를 추적하기 때문에 추적(tracing) 가비지 컬렉터라고도 부릅니다.


예시
a) 모든 객체들의 marked bit들을 false로 설정

b) 도달할수 있는 객체엔 true 마크

c) 도달할 수 없는 객체는 힙에서 정리됨


마크앤스윕 알고리즘의 장점
이건 순환 참조 문제를 제어할 수 있을 뿐만 아니라 순환할 경우에도 무한루프도 없습니다.

역주
순환참조: 단순한 레퍼런스 카운팅 방식의 공유 포인터를 사용할 경우 발생하는 누수 문제. c++의 shared_ptr이나 스위프트의 동적할당을 참고할것

게다가 알고리즘 수행중에 추가적인 오버헤드도 별로 없어요.


마크앤스윕 알고리즘의 단점
마크앤스윕 알고리즘의 주요 단점은 가비지컬렉션 수행중엔 프로그램의 실행이 잠시 중단된단겁니다.

다른 단점은 마크앤스윕이 프로그램에서 여러번 실행되면 도달 가능한 객체가 많은 미사용 메모리 영역들로 인해 분리된다는 것입니다.

아래 그림을 보시면 더 이해가 잘될겁니다.

여기에서 하얀 블럭은 free된 메모리를 나타냅니다. 회색 블럭은 도달가능한 객체가 갖고있는 메모리고요.
이제 이 free 세그먼트들은 크기가 다양합니다. 저 free 세그먼트 5개의 크기가 각각 1, 1, 2, 3, 5라고 쳐보죠.
이제 10 단위의 메모리를 생성해야 하는데요. 그럼 이제 연속적인 형태의 블럭에만 메모리를 할당할 수 있다고 가정해봅시다. 사용 가능한 메모리공간은 12 단위지만, 객체의 생성은 불가능합니다. OutOfMemory가 뜨죠.
이 문제는 “단편화(Fragmentation)”라고 불립니다. “조각들(fragments)로 사용가능한 메모리가 있지만 그 공간을 활용할 수가 없는거죠
그래서 이 단편화된 것들을 하나의 뭉치로 감출할 필요가 있습니다. 모든 free 메모리 블럭을 하나의 큰 블럭으로 합치기 위해서 메모리 항목들을 셔플하는거죠.

이제 다시 위의 예시를 살펴보면, 간결화(compaction) 후 12 단위의 연속적인 free메모리 블럭을 가지게 되므로, 10 단위의 객체도 할당을 할 수가 있게 됩니다.
