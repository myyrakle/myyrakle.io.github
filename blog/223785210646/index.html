[PostgreSQL] Unlogged Table

  • 2025.03.06
  • ::
  • 2 min read
#PostgreSQL
[원본 링크]

Unlogged Table은 높은 쓰기 성능이 필요한 예외사항을 위해 제공되는 곁가지 같은 기능이다.
"로그를 남기지 않는" 특수한 형태로 테이블을 구성하기 때문에, 일반적인 테이블보다 빠르다.

여기서 말하는 로그는 WAL을 말한다. 잘 모른다면 별도 포스트를 참고한다.
https://blog.naver.com/sssang97/223233122052




Trade-off

알다시피 모든 데이터베이스는 로그에 기반해서 데이터를 저장한다.
장애상황에 대응하고 일관성 있는 데이터 구조를 유지하기 위함이다.

Unlogged Table은 이런 로그가 저장하지 않아서 좀 더 빠른 대신에 데이터가 언제든 유실되거나 깨질 수 있다는 단점이 있다.

그래서 데이터 무결성이 크게 중요하지 않은 캐싱, 임시 데이터 로드, ETL 등에 유용하다고 주장하곤 한다.




사용법

특별한 사용법이 따로 있지는 않다. 그냥 테이블 만들떄 UNLOGGED 키워드 하나만 넣어서 생성하면 된다.

CREATE UNLOGGED TABLE unlogged_table ( 
    key varchar(100) PRIMARY KEY, 
    value TEXT 
);




성능 비교

그냥 key value 테이블 하나씩 만들어서 우겨넣고 데이터를 밀어넣어보겠다.

CREATE UNLOGGED TABLE unlogged_table ( 
    key varchar(100) PRIMARY KEY, 
    value TEXT 
);

CREATE TABLE logged_table ( 
    key varchar(100) PRIMARY KEY, 
    value TEXT 
);

기본 테이블에 1000만개 정도를 때려넣었을 때는

\timing

insert into logged_table(key, value)
select generate_series::TEXT, 'foo'::TEXT from generate_series(0, 10000000);

25초 정도가 걸렸고

UNLOGGED Table에 1000만개를 넣었을 때는

\timing

insert into unlogged_table(key, value)
select generate_series::TEXT, 'foo'::TEXT from generate_series(0, 10000000);

그거보단 적은 18초 정도가 걸렸다.

불안정성을 담보로 한 것치곤 쓰기 성능이 아주 크게 차이가 나지는 않았다. 내 하드웨어 성능이 썩 괜찮은 편이긴 한데, 그걸 감안해도 극적인 정도는 아닌 것 같다.

그리고 이건 읽기 성능은 거의 똑같다. 쓰기 성능에서만 유의미한 차이가 난다.




결론

이걸 Redis 같은 인메모리 DB 대용으로 쓸 수 있다는 주장도 있다만, 사실 이건 동의하기 어렵다.
Redis는 읽기 레이턴시를 최소화하는 것에 주 목적이 있는데, UNLOGGED 쓴다고 해서 읽기 성능이 좋아지는건 아니기 때문이다. 결국 이건 디스크 저장이다.

내 보기엔 레이턴시에 민감하지 않은 캐싱 DB로 쓰고 싶다면 그냥 기본 테이블로 만들어도 충분하다.

이 기능의 가치는 임시 데이터를 빠르게 올려서 쓰는 것 밖에는 없다.
