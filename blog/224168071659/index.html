Kong: API Gateway

  • 2026.02.02
  • ::
  • 3 min read
#Proxy
[원본 링크]

Kong은 API Gateway의 주요 구현체 중 하나다.
특이하게도 Lua 기반으로 만들어져있다.

상업용 오픈소스라서 GUI나 편의성 수준은 꽤 좋지만 라이센스에 문제가 좀 있다.
코어 모듈은 오픈소스인데, 실제 운영에 필요한 모니터링 등의 보조도구는 엔터프라이즈 버전으로만 사용할 수 있다. 사실상 Cloud-Only라고 봐도 된다.

그래서 무료로 자유롭게 사용하길 원한다면 큰 결격사항이 있다. 요즘은 대체재로 envoy가 많이 주목받더라.




비용

https://konghq.com/pricing

싸진 않다. 비용 때문에 istio나 envoy로 집단 이주하는 경우가 많이 보인다.

크게 프로비저닝 모드, 서버리스 모드, Self-Host가 있다.

프로비저닝 모드는 상당히 비싸다.
Control Plane 하나당 월별 200달러를 뜯는다.
그리고 트래픽에 대해서 큰 요금을 부과하는데, GB당 0.15달러다.
한달에 1TB만 발생해도 150달러가 부과되는 것이다.

AWS에서 비싼 편에 속하는 NAT 통신비용이 GB당 0.059달러, IGW 외부전송이 GB당 0.12달러 정도인걸 생각하면 확실히 비싸다.

서버리스 모드는 Control Plane 하나당 월별 25달러를 뜯는다.

Self-Host 모드는 정해진 비용이 없다. 저 회사랑 상의해서 정해야 한단다.




Cloud로 시작해보기

Konnect라는 Cloud를 통해서 서비스를 제공하는데, 상기한 이유들 때문에 거의 반강제로 Cloud를 사용해야 한다. 적당히 가입하면 바로 다음과 같은 대시보드에 진입하게 된다.

게이트웨이가 하나 기본적으로 만들어져있는데


엔드포인트도 열려있으나, 라우팅된 서버가 없기 때문에 동작하지는 않는다.

저걸 동작하게 만들려면 Route를 추가해야 한다.
UI에서 바로 추가해서 사용할 수 있다.

실제로 라우팅될 HTTP 서버와, 최종 엔드포인트의 라우팅 경로를 지정할 수 있다.
참고로 Route Path는 그냥 넘기기만 하는게 아니라 Path Rewriting까지 자동으로 해준다.

다음과 같은 간단한 API 서버를 하나 붙여보겠다.



전체 API 엔드포인트를 그대로 넣고, /a 하위 경로로 매핑되도록 했다.
저렇게 해서 배포하고 나면

/a 경로로 진입했을 때 / 경로로 라우팅을 해줄 것이고

/a/a로 진입했을 때는 /a 경로로 라우팅을 해줄 것이다.

인증키 구성

API Gateway를 쓰는 목적 중 하나는 여러개의 마이크로서비스에 대해서 단일 인증 구간을 관리하는 것이다.
Kong도 이를 위한 기능을 손쉽게 제공한다.

위와 같이 키를 추가하면, 즉시 키 기반의 인증 가드가 걸린다.
참고로 Username은 논리적인 이름일 뿐이고, Key Name과 Key가 한 쌍의 HTTP 헤더로 들어가는 것이다.


그러면 이제는 단순 접근이 막힐 것이고


이렇게 헤더로 키를 넘겨야만 통과가 된다.




플러그인

Kong은 플러그인을 통해서 많은 핵심 기능을 제공한다.
CORS 처리, Rate Limit, JWT 인증, 세션 관리 등.

플러그인을 활성화하면 Rate Limit도 간단하게 작동시킬 수 있다.

이건 하루 100건, 시간당 5건을 초과할 경우 Limit을 거는 옵션이다.
