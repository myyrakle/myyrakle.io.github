[Git] 브랜치

  • 2020.01.19
  • ::
  • 3 min read
#Git
[원본 링크]

기본적으로, 프로젝트는 협업으로 이루어진다.

그러다보면 여러 사람들이 작성한 코드, 이슈별 코드, 그리고 기능별 코드들을 분리해서 관리할 필요가 있는데...

브랜치는 그런걸 위한 기능이다.

브랜치(branch)는 단어의 뜻 그대로, 분기를 나눈다.
그럼 각 개발자들은 자신의 브랜치를 사용해서 푸시하고, 운영자는 그걸 잘 통합해주면 된다.
그리고 개발자 단위가 아니라, 특정 기능, 버그 픽스 단위로 브랜치를 쪼개서 관리할 수도 있다.

깃헙 페이지를 보면 항상 master라고 뜨는게 있고, 이전에 push 등을 할때 master를 쓴 적도 있는데.
이 master도 사실 브랜치 중의 하나다. 깃헙에서 자동으로 생성해주는 기본 브랜치일 뿐이다.

먼저 테스트를 하면서 살펴보자.
Github에서 TestGit이라는 레포지토리를 만들고, clone으로 그걸 받아왔다.

기존에 존재하던 파일은 대충 이렇다.

그럼 이제 하나씩 익혀보자.


브랜치 확인
그냥 git branch를 치면 현재 존재하는 브랜치들을 받아볼 수 있다.

마스터 하나가 있다.


브랜치 생성
브랜치 생성도 간단하다.
그냥 아래와 같이 치면 된다.
git branch 브랜치명

bugfix라는 이름의 브랜치를 추가했다.


브랜치 전환
근데 저렇게 만들어두기만 해서는 쓸데가 전혀 없다. 여전히 브랜치는 master인 상태이기 때문이다.
해당 브랜치로 전환하려면, checkout 명령을 쓰면 된다.
git checkout 브랜치명

잘 바뀐걸 볼 수 있다.

참고로 checkout에 -b 플래그를 주면 브랜치를 생성하면서 전환할 수 있다.

그럼 이제 뭔가 변경사항을 좀 줘보자.
bugfixed라는 쓸데없는 텍스트 파일을 만들고, 추가하고, 커밋했다.

뭐 잘 된다.

한번 푸시도 해보자.

잘 된다!

그리고 깃헙에 가보면

브랜치가 추가된걸 확인할 수 있다. 브랜치 모드를 바꾸면 변경된 코드들도 전부 확인할 수 있고.


브랜치 병합하기
근데 저러기만 해서야 뭐 되겠는가?
변경된 사항을 다시 한곳에 뭉쳐야 뭐가 되지.

어차피 메인으로 관리되는 브랜치는 master다.
master로 돌아가, bugfix에서 추가되거나 변경된 사항을 병합해줘야 한다.

여럿이서 협업할 경우엔 아마 pull을 통해 다른 브랜치 작업을 가져온 다음에 병합을 시도해야 할것이다...

명령어는 아래와 같다.
git merge 병합할브랜치명

그럼 이렇게 업데이트가 된다.

제대로 됐다면 뭐 그냥 푸시하면 된다.


브랜치 삭제하기
일회용 이슈성의 브랜치인 경우, 작업이 끝나면 깔끔하게 삭제하는 게 도리일 것이다.
뭐 그냥 이렇게, branch 명령에 -d 플래그를 주면 된다.
git branch -d 브랜치명

여기선 아주 기본적인 사항만 다뤘지만, 이거 말고도 기능이나 주의점이 이것저것 많다.
나머지는 다음 포스팅에서 다루겠다.
