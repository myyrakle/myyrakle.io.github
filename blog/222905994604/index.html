[원본 링크]

github 프로젝트는 말 그대로 프로젝트 관리 기능이다.

예전에는 레포지토리 하나에 종속된 채로만 기능했고, 여러모로 기능이 많이 딸려서 쓰다 던진 기억이 있는데.. 간만에 보니까 완전 다 갈아엎었더라.
지금은 레포지토리와 별개로 관리되고. 여러모로 잘 다듬어진 느낌이 난다.




프로젝트 생성

프로젝트를 관리할 계정이나 조직으로 접속해, 프로젝트 생성을 클릭한다.

그럼 이렇게 생성과 동시에 바로 진입이 될 것이다.

그리고 이름만 바꿔주면 된다.




프로젝트 설정

기본설정에서 바꿀 것이 있다면, 오른편에 설정창으로 이동해서 건드리면 된다.

가시성을 조절하고 싶다면 저기 아래 Private 값을 바꿔준다.
기본값이 Private이라서 이 상태로는 초대한 사람만 프로젝트를 볼 수 있다.
내 경우에는 그냥 별볼일없는 오픈소스라서, Public으로 열어놨다.

manage access 탭으로 가면, 함께 작업할 사람들을 모집할 수 있다.

그리고, 프로젝트는 레포지토리와 별개로 존재한다고 했다.
하지만 그렇다고 해서 레포지토리와 관계가 없는 것은 아니다.

레포지토리의 Projects 탭으로 이동하면, 연관있는 프로젝트를 연결할 수 있다.
그냥 겉으로 묶어주는거지 특별한 기능이 생기지는 않는다.




뷰 관리

뷰는 태스크 목록을 보여주는 방식을 정의한다. 하나의 프로젝트에는 여러개의 뷰를 둘 수 있다.
유의할 점은 보여줄 방식을 바꿀 뿐이지, 그 내부의 시트 데이터는 전부 하나를 공유한다는 것이다. 필터를 다르게 걸고, 노출 컬럼을 달리할 수 있을 뿐이다.

New View 버튼을 누르면 뷰를 추가할 수 있고, 이름을 클릭하면 이름을 바꿀 수 있다.

이런식으로 메인태스크용 뷰와 버그 수정 관리용 뷰를 따로 두는 식으로 사용할 수 있겠다.




draft 생성

draft는 노션이나 다른 도구에서도 사용되는 용어인데, 하나하나의 이슈 단위라고 할 수 있겠다.
여기서는 뷰에 생성되는 한줄한줄이 draft다.

  • 버튼을 누르고 텍스트를 치는 간단한 작업으로 바로 생성할 수 있다.

그럼 이렇게 추가되는데

담당자나 진행상태 등의 값을 관리할 수 있고

별도의 필드들도 다 추가해 사용이 가능하다.

본문은 일반 블로그처럼 작성이 가능하다.

이걸 오른편의 Convert To issue 바로 이슈로 박아넣을 수도 있다.




이슈 연동

레포지토리에 존재하는 이슈나 풀 리퀘스트를 바로 가져와서 연동해서 링크할 수도 있다.

상태 컬럼만 관리할 수 있고, 원본 이슈에 영향을 주지는 않는다.

그렇다.

가볍게 쓰기는 꽤 괜찮은 것 같다.
성능도 쾌적하고, 심플하고 강력하다.
