아마 이게 위버를 만든 주 목적 중 하나이지 않나 싶다. GCP 유저 늘리기...

GCP의 쿠버네티스 서비스에 분산 애플리케이션을 띄우는 기능을 기본으로 제공한다.

다른 클라우드 기능이나 온프레미스 기능도 추후에 제공될지는 잘 모르겠다. 현재로서는 고려하고 있지 않은듯.




weaver gke CLI 설치

weaver cli를 사용해서 자동화된 배포를 시도하려면, 또 전용 도구를 하나 깔아줘야 한다.

go install github.com/ServiceWeaver/weaver-gke/cmd/weaver-gke@latest




GCP CLI 설치

gcp cli가 없다면. 아래 링크에 들어가서 적절히 깔아준다.
gcloud CLI 설치 | Google Cloud


플러그인도 하나 깔아주고

gcloud components install gke-gcloud-auth-plugin

계정 정보도 적절히 넣어주면 된다.




밑준비

루트 URL로 헬스체크용 API를 하나 파줘야 한다.

    // Health Check
	http.HandleFunc("/", func(writer http.ResponseWriter, request *http.Request) {
		fmt.Fprintf(writer, "OK")
	})

이게 없으면 컨테이너의 건강함을 알 수가 없어서 재배포가 끝나도 구버전을 내리질 못한다.




배포하기

먼저 toml 파일을 적절히 수정해줘야 한다.

그리고 windows/macOS 환경에서는 배포를 시도하기에 한계가 있다.
어차피 쿠버로 배포하려면 실행파일을 리눅스용으로 말아야 하기 때문이다...

그리고 gke 옵션을 추가한다.

[serviceweaver]
name = "weaver-start"
binary = "./weaver-start"
# binary = "./weaver-start.exe"

[gke]
regions = ["asia-northeast3"]
public_listener = [
  {name = "weaver-start", hostname = "weaver.myyrakle.com"},
]

실제로 배포할 리전과, 퍼블릭 리스너를 정의한다.
퍼블릭 리스너를 정의하지 않으면 private 리스너만 열려서 내부 통신만 가능해진다.

그리고 gke deploy 명령으로 실행해주면 된다.

weaver gke deploy weaver.toml

그럼 한세월동안 클러스터 구성을 시작한다.


유의할 점은, 중앙제어용 config 클러스터를 추가로 띄운다는 것이다.
현재는 us-central1 리전을 고정으로 띄운다.

그리고 설정과 배포가 끝나면, 노드도 알아서 잘 띄워줄 것이다.

서비스 상태는 gke status 명령으로 확인할 수 있다.

따로 퍼블릭 리스너를 설정하지 않으면 위와 같이 private 로드밸런서로 구성된 private 엔드포인트만 만들어지고

퍼블릭 리스너를 설정하면 이런 식으로 된다.





public 엔드포인트 호출

그냥 간단하게 잘 도나 테스트만 해보고싶다면, 요청 헤더에 도메인 정보만 넣고 public IP에 쏴보면 된다.

curl --header "Host: 도메인" http:...

제대로 쓰려면 도메인 붙여서 쓰면 더 좋고

근데 뭔가 SSL 설정은 자동으로 해주긴 해주는데... SSL 인증서 프로비저닝이 한세월 걸린다.


24시간까지 기다려보란다.
내 경우에는 2시간쯤 걸린 것 같다.
AWS는 몇분이면 되던데... 좀 아쉬운 속도다.




private 엔드포인트

VPC 내에서만 사용할 private 엔드포인트는 "서비스명.리전명.serviceweaver.internal"의 구조를 가진다.

그래서 이렇게 쓰면 된다고 안내하긴 하는데..

모르겠다. 잘 안된다.
내부 로드밸런서에 쓰는 envoy에서 발생하는 오류일텐데, 아직 만들다 말아서 그런건지 뭔지는 잘 모르겠다.
