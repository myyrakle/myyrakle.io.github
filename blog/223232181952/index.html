LSM 트리

  • 2023.10.09
  • ::
  • 4 min read
#데이터베이스
[원본 링크]

LSM 트리는 Log Structured Merge Tree의 준말로, 로그와 같이 사이즈가 큰 데이터를 트리 형태로 저장해서 인덱싱할 수 있게 해주는 자료구조다.

Patrick O'Neil외 3명에 의해 1996년에 고안되었다.

응용 수준에서 자주 쓰진 않고, 데이터베이스 같은걸 만들때 주로 사용된다.
RDB에서 지배적으로 사용되는 전통적인 인덱스 구조인 B-Tree와 대비되는 편이다.




LSM 트리의 구조와 원리


LSM 트리는 일반적인 트리와 다르게 몇개의 레이어로 구성된다. 최초로 제안된 형태는 2단계이나, 일반적으로는 위 그림과 같이 3단계로 구성된다.
파일 사이즈가 너무 클 때를 고려해서 중간 sort 레이어를 하나 더 추가한 것이다.

여하튼 기본적인 저장 원리는 이렇다.

  1. 빠르고 작은 메모리(RAM)에는 진짜 트리 형태로 인덱스 데이터를 관리한다. 이를 memtable이라고 부르며, 레드블랙 트리, skiplist 등으로 구현될 수 있다.
    (level 0)
  2. 느리고 큰 디스크에는 정렬된 배열의 형태로 데이터를 관리한다. 이를 Sorted String Table(SST)이라고도 부른다.
    (level 1, 2)

쓰기 동작 원리는 이렇다.

  1. 처음 데이터를 삽입/수정할 때는 작고 빠른 memtable에 데이터를 저장한다.
  2. 만약 memtable의 크기가 일정치(몇 MB)를 초과하면 그걸 파일로 저장해서 디스크에 보관한다.
  3. 파일이 생성되는 과정은 항상 append only로 이루어지기 때문에 seq I/O로 돌아서 매우 빠르다.

읽기 동작 원리는 간단하다.

  1. 읽기 요청이 들어오면 memtable에서 키를 찾는다.
  2. 키가 memtable에 없다면 디스크의 최신 세그먼트부터 SS Table을 뒤져서 값을 찾는다.
  3. 이 읽기 성능은 O(N)로 이루어지기 때문에 파일이 커짐에 따라 성능이 심각하게 저하될 수 있다.
  4. 때문에 성능 보완을 위해 여러가지 보조적인 알고리즘을 활용한다.



컴팩션 (Compaction)

LSM은 계속해서 파일을 append하는 구조라고 했다.
때문에 파일이 늘어날때마다 read 시간이 선형적으로 늘어나는데, 그걸 좀 줄이기 위한 방법이 컴팩션이다.

그래서 LSM에 기반한 대부분의 DB 시스템에서는 백그라운드 프로세스에서 컴팩션을 돌려서 여러개의 SS Table 파일을 병합하는 과정을 거친다.

애초에 각 파일들이 정렬된 상태로 저장되어있기 때문에 이 병합은 매우 빠르게 이루어지고, 이 동작형태가 merge sort와 비슷하다고 해서 Merge Tree라고 불리는 것이기도 하다.

아무튼 이 컴팩션 과정에서 중복된 키나 삭제된 키는 제거된다.

LSM Tree vs B-Tree

LSM 트리가 새로 떠오르는 형태의 구조라고 해서 B-Tree보다 우월하다거나 한건 아니다.
그러면 다 갈아엎고 B-Tree를 버렸겠지...

각각 장단점이 있는데, 간단하게 말하면 대체로 LSM은 write가 빠르고, B-Tree는 read가 빠르다고 할 수 있다.
write에 최적화된 NoSQL 시스템들이 대부분 LSM을 차용하는 이유이기도 하다.

디스크 저장 효율도 LSM이 좋은 편이다. LSM이 순차적으로 저장돼서 디스크가 낭비없이 꽉꽉 차는 반면에, B-Tree는 파편화돼서 빈 공간이 많이 생긴다.

동시성 제어 측면에서는 LSM이 느슨한 구조를 가져서 불리한 부분이 있다.
B-Tree는 한 키가 동시에 한 곳에만 존재한다는게 보장되는데, LSM은 여러 파일에 퍼져있을 수 있기 때문이다. 해서, 강력한 트랜잭션 수준을 보장해야 할 때는 B-Tree가 유리한 부분이 많다.




사용례

응용 수준의 사용례는 적지만, 시스템 수준에서는 매우 많이 사용된다.

Apache AsterixDB, Bigtable, HBase, LevelDB, Apache Accumulo, SQLite4, Tarantool, RocksDB, WiredTiger(MongoDB), Apache Cassandra, InfluxDB, YugabyteDB, ScyllaDB, and CockroachDB 등... 비교적 근래에 탄생한 DB 시스템들은 대부분 LSM 트리로 인덱스 시스템을 구현했다.
이거 말고도 많다.
