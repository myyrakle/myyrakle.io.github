[Git] 시작

  • 2019.11.08
  • ::
  • 3 min read
#Git
[원본 링크]

이전 포스팅에서 로컬 저장소와 원격 저장소에 대해 언급했다.
원격 저장소의 종류는 뭘 골라도 상관없지만, 최소한 하나는 있긴 있어야 한다.

나는 Github를 사용하도록 하겠다.

먼저 테스트에 사용할 레포지토리를 하나 만든다.

일단은 비워놓겠다.

이제 로컬 저장소를 마련해보자.
폴더부터 적당히 잘 만든다.

이제 Git을 써야하지 않겠는가?

저 폴더 위치 안에서 마우스 오른쪽 버튼을 누른다.

2가지가 있다.
좀 편하게 볼수있는 GUI.
그리고 Bash라는 이름의 터미널.

일단 기본이 중요하니 터미널로 먼저 열어보자.

그럼 이렇게 터미널이 뜬다.

가장 먼저 해야할 건 닉네임과 이메일 주소를 입력해주는 거다.
안하면 commit 못하게 한다.

로컬 저장소를 만드는 것엔 2가지 방법이 있는데, 처음부터 만드는 init이 있고.
기존에 존재하던 원격저장소 코드를 가져오는 clone이 있다.

init부터 사용해보자.

이러면 .git이라는 이름의 설정파일 폴더가 생긴다.

근데 이대로는 아무것도 없고 공허하지 않은가.
소스파일을 뭘로든 하나 만들어서 넣어보자.

나는 따로 뭘 열기가 귀찮아서 Vim을 썼지만, 뭘 써도 상관없다.

이제 이걸 .git에 등록하면 된다.
커맨드는 add 파일명이다.

이러면 추가가 되긴 하는데.

실제로는 반영이 된게 아무것도 없다.
저렇게 추가를 한 것들은 git commit 커맨드로 커밋을 해줘야 제대로 저장이 된다,

그럼 아까 설정했던 에디터로 변경내역이 보인다.


vscode의 경우, 저기서 new file 저 부분의 주석(#)을 지우고 닫으면 커밋이 수행된다.


됐다.



원격 저장소에 저장

이제 저것들을 제대로 저장해야 하지 않겠는가?

2가지 방법이 있는데.
remote란 원격저장소 관리자를 만들어서 저장하는 것과,
그냥 링크를 때려박아서 올리는 것이다.
먼저 무작정 써보도록 하겠다.

그리고 push를 하기 전에 먼저 pull이란 걸 해야한다.
push는 기본적으로 git에 저장된 정보를 바탕으로 원격저장소에 통째로 덮어쓴다!
때문에, 원격저장소에 로컬에 없는 파일이 있고, git도 그걸 모른다면 그 파일은 push 시 증발할 위험이 있는 것이다. 이런 위험 때문에 그냥 push는 저런 상황에서 에러를 뱉지만.
push -f는 강제로 처리한다. 그래서 이건 쓰지 않는게 좋다.

여튼 pull은 원격 저장소의 내역을 가져와서 로컬 저장소를 동기화시키는 건데.
나는 깃헙 레포지토리도 그냥 손으로 만들었고, 로컬 저장소도 그냥 init으로 만들었다.
이 두개의 저장소는 아무 관련이 없던거였으니... 이런 처리를 해줘야 하는 것이다.
그래서 가급적 pull을 먼저 해주는 것이 좋다.

git pull 링크 master --allow-unreleated-histories

저 기다란 플래그는 이런 상황 말고는 쓸일이 없다.

이제 제대로 업로드를 해보자.

git push 링크 master
링크에는 아까 만들었던 Github 레포지토리의 링크를 넣는다.
그리고 master는 지정할 branch다.

이러면


로그인하라고 뜬다.
당연하다.


그럼 이렇게 했다고 뜨는데.

깃헙에 가보면

실제로 반영이 되어있다.



remote를 통한 원격저장소 관리

위에서는 링크를 일일이 때려박았지만, 이를 위한 편의기능이 존재한다.
remote다.

remote는 다음과 같이 추가한다.

git remote add 이름 링크

혹시 링크를 잘못 쳤다면 git remote remove 이름 으로 지우고 다시 만들거나,
**git remote set-url 이름 링크 **로 수정하면 된다.

여튼 이렇게 remote를 만들면, 귀찮게 계속 링크를 쓸게 아니라 remote 이름으로 대체할 수 있다.

foo.rs란 파일을 하나 더 만들어보자.

그럼 이제 이렇게 깔끔하게 push 커맨드를 날릴 수 있다.

실제 저장소에도 잘 들어갔고.
