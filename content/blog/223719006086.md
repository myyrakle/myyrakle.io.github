Github Action은 훌륭하고 사용감 괜찮은 CI/CD용 서비스지만, 그냥 기본 제공 호스트만 써서는 한계가 있을 때가 많다.  

1. 프로비저닝 시간이 좀 걸리고  

2. 서버 성능도 그렇게 좋은 편은 아니고  

3. 요금도 많진 않지만 조금 나오는데다  

4. 온프레미스 머신에 직접 접근해서 처리하는게 불편하기 때문이다.  

<br>

<br>

그래서 추가적인 요구사항이 있다면 Github Action에서 사용하는 Self-Hosted Runner를 사용해서 빌드 환경을 확장할 수 있다.  

이렇게 설정할 경우에는 트리거와 Github 코드 복제까지만 Github에서 제공해주고, 각 머신에서 독립적으로 동작한다. 가상화도 없다. github action cache 같은 부가적인 기능들도 활용하는데 제한이 있다.  

그리고 가상화 없이 해당 머신에 직접 실행되는 것이라서 빠르긴 한데, 보안이나 빌드 리소스 cleanup에 대한 고려가 조금 필요할 수 있다.  

<br>

<br>

## Daemon 구성하기 (Linux)

github action runner는 그냥 실행하기에는 충분하지만, 지속적으로 사용할 수 있는 데몬에 대한 구성은 사전 제공하지 않는 것 같다.  

머신이 재부팅되어도 알아서 잘 뜨고, 항상 살아있게 만들려면 systemd를 활용하는게 간편하다.  
[https://blog.naver.com/sssang97/223374364520](https://blog.naver.com/sssang97/223374364520)  

설정파일 만들고  

```javascript
sudo vi /etc/systemd/system/action-runner-1.service
```

![](/images/img_1f74c6d7562efd0e.png)

```javascript
[Unit]
Description=Github Action Runner

[Service]
Type=simple
Restart=on-failure
RestartSec=1
User=user
WorkingDirectory=/home/user/actions-runner
ExecStart=/home/user/actions-runner/run.sh

[Install]
WantedBy=network-online.target
```
적당히 이렇게 넣어준다.  
User가 현재 사용중인 유저 정보가 되어야 한다. root는 안된다.  

그리고 활성화하면  

```javascript
sudo systemctl enable --now action-runner-1.service
sudo systemctl status action-runner-1.service
```
