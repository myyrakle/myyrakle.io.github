C#은 거의 순수한 객체지향적인 언어이고, 때문에 모든 일은 클래스 단위에서 이루어진다.  

C언어처럼 전역에 변수놓고 함수놓고 그런건 안된다. 전역에 올수 있는건 오직 클래스를 비롯한 타입의 선언 뿐이다.  

C#은 대부분의 언어가 그렇듯이 이런 식으로 클래스를 정의한다.  

class 클래스명 : 상속할타입  
{  
접근제어자 타입 변수;  
접근제어자 타입 프로퍼티 { 프로퍼티 몸체 };  
접근제어자 타입 메서드(파라미터) { 메서드 몸체 }  
접근제어자 클래스명(파라미터) { 생성자 몸체 }  
접근제어자 ~클래스명() { 소멸자 몸체 }  
......등등  
}  

변수나 메서드는 굳이 설명하지 않겠다.  

<br>

**프로퍼티**  
C#에는 프로퍼티라는 게 있는데, 게터/세터 메소드를 추상화한 것이다.  

이런식으로 쓴다,  

여기서 사용된 get과 set, value는 상황별 키워드이다. 다른데서는 식별자로 쓸 수 있단거다.  
get 블럭은 반환값이고 set 블럭은 집어넣을 값을 명시한다. value는 대입값으로 들어오는 파라미터를 추상화한 것이다.  
이렇게 하면 위처럼 name 변수에 대한 프로퍼티를 만들어서 변수처럼 사용할 수 있다.  

이게 왜 필요한걸까? 어차피 저렇게 공개해서 쓸거면 그냥 public 변수를 쓰는게 낫지 않나?  
근데 사람들은 이게 오히려 뭐 추상화가 잘된다나 뭐라 하는데 나는 잘 모르겠다.  
그래도 get과 set 여부를 조정할 수는 있다. set을 아예 빼버리면 읽기전용이 되는것이다.  

그리고 버전 3.0부터는 더 편리한 프로퍼티 자동구현 기능이 생겼다.  
get과 set의 구현을 지워버리고 변수를 없애버리면, 프로퍼티가 알아서 변수를 만들어 숨겨놓고 갖다쓴다.  

훨씬 편리해졌다.  
그리고 이 기능의 기발함은 get이나 set에 접근제한을 걸 수 있다는 것이다.  

<br>

이러면 값의 할당은 클래스 내부에서만 지정할 수 있다.  

게다가 버전 6.0부터는 여기다 디폴트값을 넣을 수도 있다.  
흠. 훌륭하다. 프로퍼티는 이걸로 끝이다.  

<br>

생성자 사용법은 자바랑 똑같다.  

상속은 굉장히 깔끔하다.  
자바처럼 extends, impl... 드럽게 타이핑쳐야 하는것도 아니고, 접근제어를 붙이는 것도 아니다.  

이렇게.  

그리고 저 base는 슈퍼클래스를 가리키는 포인터다. 자바의 super와 동일하다 할 수 있다. 저렇게 쓰면 슈퍼클래스의 생성자를 호출하는거다.  
자신을 가리키는 키워드로는 this가 있다.  

this로도 저런식으로 생성자를 울궈먹을 수 있다. 생성자 위임이라고 한다.  
생성자() : this() {...}  

소멸자는 객체가 소멸할때 나오는건데 딱히 직접 쓸데는 없다. 그러니까 그냥 넘긴다.  

<br>

아참 접근제어자는 무려 이렇게 6개나 있다.  
private  
public  
protected  
internal  
internal protected  
private protected (since C# 7,2)  
근데 위에 두가지 말고는 쓸일 없으니 잊어버려도 된다.  

<br>

그리고 객체를 생성할때 중괄호로 변수나 프로퍼티명을 지정해서 직접 집어넣을 수도 있는데, public 멤버에만 해당된다. 생성자 파라미터와 동시에 사용될 수 있다. 이건 버전 3.0부터 나온 기능이다.

Static  
static에 대한 개념은 C++이나 자바와 동일하다.  
그런데 static멤버만을 가질 수 있는 static class가 존재한다.  
바로 위의 이미지처럼 class 선언 앞에 static만 붙이면 된다. 이 정적 클래스에 비정적 멤버가 선언되면 빨간줄이 쳐진다.  
개인적으로 Main 메소드의 클래스는 정적클래스로 지정하는게 좋다 생각한다.  

그리고 static멤버만을 따로 묶어서 초기화하는 정적 생성자란게 있다.  
static 클래스명()  
{  
초기화작업 수행  
}  
이렇게 쓰면 된다. 파라미터는 사용할 수 없으며 정적 멤버의 초기화만을 담당한다.

구조체  
클래스 선언에서 class만 struct로 바꾸면 구조체가 된다.  
기본적인 사용법은 클래스랑 같은데  
몇가지 제약사항과 차이점이 있다.  

일단 클래스는 참조타입인데다가 가비지컬렉터가 처리를 해준다.  
근데 구조체는 값 타입이라 그냥 로컬을 벗어나면 날라간다.  

클래스는 힙에 할당된 레퍼런스를 통해 작동하기 때문에 주소만을 복사하는 얕은복사가 기본동작이다.  
구조체는 깊은복사다.  

클래스는 생성할때 반드시 new를 쓰는데,  
구조체는 써도되고 안써도된다. 작동에 차이는 없다. new 붙인다고 동적으로 할당하거나 그러진 않는다.  

구조체는 매개변수 없는 생성자를 선언할 수 없다.  
구조체의 생성자는 모든 필드를 초기화해야하는 의무가 있다.

인터페이스  
인터페이스는 자바랑 별차이 없다.  
변수가 없고 메서드의 정의가 없는, 껍데기만을 가지는 타입이다. 때문에 이거만으로는 객체를 생성할 수 없다. 뭐가 있어야 생성을 하지  
아 귀찮다. 어차피 여기까지 봤으면 다 알테니 요점만 쓰겠다.  
-메서드 구현이 없는 클래스  
-객체 생성 불가  
-변수 소유 불가  
-접근제어 불가  
C#도 자바처럼 다중상속을 금지하지만,  
인터페이스에 한해서만 다중상속이 된다. 뭐 들어있는게 없으니 충돌할것도 없지.  

<br>

추상클래스  
추상클래스는 인터페이스와 클래스를 섞어놓은거다. 구현은 가능한데 메서드 선언만 남겨놓고 저걸 상속받을 클래스한테 전체적인 구현을 유예할 수 있다.  
디자인패턴에 유용하게 쓰인다.  

여튼 대략적인 특성은 이렇다.  
-메서드 구현을 할 수 있다. 안할 수도 있다. 맘대로다  
-객체 생성 불가(동일)  
-변수 소유 가능  
-접근제어 가능  

abstract class 클래스명  
{  
public abstract void 추상메서드명();  
...  
}  
이런식으로 사용한다.
