Double Dispatch는 다형성을 활용한 변형 패턴 중 하나다.  

Dispatch를 2중으로 한다는 건데, 여기서 Dispatch는 Dynamic Dispatch - 다시 말해서 메서드 오버라이딩을 통한 함수 수준의 다형성 동작이다.  

Dynamic Dispatch는 기본적으로 특정 객체의 실제 타입에 따라서 실제 동작 메서드를 다르게 하는 것이고, 하나의 객체의 상태에 따라서만 분기해도 충분한 경우에는 이것으로도 충분하다.  
Double Dispatch는 객체 2가지의 타입의 조합에 따라서 동작을 더 복잡하게 제어할 필요가 있을 때 사용된다.

## Single Dispatch

코드로 간단한 예시를 하나 들어보겠다.  
언어는 Rust이긴 한데, 이해에 어려움은 없을 것이다.  

다음 코드는 Person이란 인터페이스를 정의하고, 각 객체들이 인터페이스를 구현하는 것으로 객체의 타입별 동작을 다르게 구현했다.  

![](/images/img_b082d2669137701d.png)

![](/images/img_efebf5343a181e99.png)

![](/images/img_03e1c9795ff8598a.png)
그러면 실제 객체 타입에 따라서 다른 동작 - 다른 로깅을 할 것이다.  

이게 일반적으로 말하는 객체의 다형성이다.  
사실 구현 내부적으로는 객체의 타입 플래그를 통해서 가상함수를 if-else로 호출하는 것이지만, 문법적으로는 객체가 능동적으로 자신의 행동을 수행하는 것처럼 깔끔하게 보인다.

## 객체 간의 복잡한 상호작용

이제 좀 더 복잡한 예시를 들어보겠다.  
사람이 다른 사람과 만났을 때의 동작을 구현하려면 어떻게 해야 할까?  

그러니까 예를 들면, 학생이 학생과 만났을 때, 학생이 교사와 만났을 떄, 교사와 교사가 만났을 때 등을 전부 다르게 구현하려고 한다.  

![](/images/img_aadbd19fe57e7adf.png)

![](/images/img_e09949c8302d071c.png)

![](/images/img_3f0483e16d91d610.png)
단순하게 구현한다면 이런 식으로 구현할 수 있을 것이다.  

하지만 이런 방식은 기본적인 객체지향 논리에서 많이 벗어난다.  
사용자의 최종 함수 인터페이스가 개별 구체 타입에 의존해버리는 것이기 때문이다.  

아니면, 동작을 단일 메서드 인터페이스로 정의하되, 해당 구현에서 타입으로 분기를 칠 수도 있다.  

![](/images/img_c003052ebf21e51b.png)

![](/images/img_2cb2f4038fa0da9f.png)
동작은 하지만, 이것도 안티 패턴에 속한다.  
다운캐스팅을 통한 타입 체크는 가독성도 나쁘조 변경이나 확장에 취약한 부분이 있기 때문이다.

## Double Dispatch

Double Dispatch는 다형성에 의한 함수 호출 분기를 2중으로 처리하는 것으로 이 문제를 해결한다.  

![](/images/img_4a726f7f04af257f.png)
예를 들면 이런 식이다.  
API 사용자가 사용할 최종 함수가 있고, 거기에서 분기를 태울 2번째 디스패치 함수가 있다.  

<br>

![](/images/img_c24fe287fa505fdb.png)

![](/images/img_ac0b5a49142f5f76.png)
그럼 각각의 객체들이 함수를 적절히 구현해주기만 하면 된다.  

사용자가 사용할 때는 처음 meet으로 객체 분기를 타고 (first dispatch), other의 개별 meet을 부르면서 2번째 객체 분기를 탄다. 그래서 double dispatch다.  
이를 통해서 2가지의 타입 분기를 다형성을 기반으로 숨겼다. 사실 if-else는 있지만, 이제 그게 코드에서 드러나진 않는다.  

<br>

![](/images/img_dd853e103bacaa1a.png)
최종 사용자는 이렇게 first dispatch 함수를 사용하기만 하면 된다.

## Double Dispatch의 한계

사실 위 예제에는 미묘한 부분이 있다.  

![](/images/img_27985a881126ae51.png)

![](/images/img_728cd2e042744764.png)
meet 함수 내에서 우항 객체를 기준으로 구체 메서드를 호출하기 때문에, 왼쪽과 오른쪽의 순서가 뒤바뀐다는 것이다. 주체가 바뀌었다.  

그래서 이건 객체 타입 간의 대칭성이 보장될 때만 이상적으로 잘 동작하는 구현 패턴이다.  
객체 간의 연산 순서가 중요한 경우에는 사용하기에 좀 애매하다.
