bisect는 문제 추적을 위한 독특한 명령어 중 하나다.  
이건 커밋 히스토리 단위로 "이진 탐색"을 수행하는 기능이다.  

시스템을 운영하다보면, 어떤 커밋에서 문제나 변경이 발생했는지 추적해야할 일이 드물지 않게 발생한다.  
문제가 명확하다면 그냥 알아서 잘 고치면 되지만, 그러기 어려운 경우에는 문제가 발생한 전/후의 코드 변경을 비교하고 결론을 도출하는 것이 가장 효과적이기 때문이다.  

커밋이 적거나, 본인이 그 변경 히스토리를 얼추 알고 있다면 문제가 생긴 커밋을 찾을 수 있지만, 커밋히스토리가 방대한 레거시 시스템의 경우에는 그러는 것도 쉬운 일이 아니다.  

bisect는 이런 상황에 사용된다. 이것 자체가 귀신같이 알아서 찾아주고 그러는건 아니고, 이진탐색하듯이 커밋 내역을 이동하면서 문제가 발생한 지점을 추적하는 상태관리를 해줄 뿐이다.  

이걸 쓰려면 우선 start 명령을 사용해서 bisect 컨텍스트에 진입해야 한다.  

```javascript
git bisect start
```

![](/images/img_faa36c8444238b07.png)

그리고 현재 커밋이 "문제가 있는" 커밋이라고 마킹해놓는다.  

```javascript
git bisect bad
```

![](/images/img_c6016dc1e21a02df.png)
이게 이진탐색의 오른쪽 끝 종료지점이 된다.  

그럼 bisect log를 통해 상태 내역을 확인할 수 있다.  

![](/images/img_cd6200a6feb11d24.png)

마찬가지로 이진탐색의 "시작점"도 정의해야 한다.  
확실히 문제가 없다고 판단되는 커밋을 하나 선택한 다음에, 그걸 "good(문제 없음)"으로 마킹한다.  

```javascript
git bisect good <커밋해시>
```

![](/images/img_3c58c31909701ae0.png)

자, 이러면 bisect로 문제를 찾을 준비가 완료된 것이다.  
이제부터는 현재 커밋에서 문제가 있다면 good, 문제가 없다면 bad를 찍어주면 된다.  

그럼 마킹을 할때마다 이진탐색 원리를 기반으로 git이 자동으로 범위를 좁히면서 스위칭을 해준다.  

![](/images/img_89a70e15de5550f1.png)
그렇게 계속 찍다보면, 결국에는 good->bad로 변화했다고 판단할 수 있는 커밋을 추론해낼 수 있게 된다.  
그러면 이렇게 결론을 내준다.  

별거 아닌것처럼 보일 수 있지만 Git에서 가장 강력한 기능 중 하나라고 생각한다.  
시스템의 복잡도가 높고 엮여있는 것이 많다면 그냥 코드를 읽는 것만으로는 문제를 추적하기 어렵기 때문이다.  

<br>

![](/images/img_8c8c45832cc3faf1.png)
그래서 판단이 다 끝났다면, reset을 사용해서 컨텍스트를 초기화하면 된다.
