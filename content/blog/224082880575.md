langgraph는 langchain이 확장된 형태의 에이전트 개발용 프레임워크다.  
구조화된 에이전트간 상호작용을 잘 지원하기 때문에, 멀티에이전트 기반의 복잡한 요구사항이 필요한 경우라면 langchain보다는 langgraph를 선택하는 것이 권장된다.  

여기서는 기본적인 사용 구조만 간단하게 다뤄본다.  

<br>

<br>

<br>

## 종속성 구성 (with Gemini)

uv를 쓰고, gemini를 쓴다고 가정한다.  

기본적으로 필요한 종속성은 이 정도다.  

```javascript
[project]
name = "langchain-test"
version = "0.1.0"
description = "Add your description here"
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
    "langchain>=1.0.0",
    "langchain-google-genai>=1.0.0",
    "python-dotenv>=1.0.0",
    "langgraph>=0.1.0",
]
```
langgraph를 쓴다고 해서 langchain을 쓰지 않게 되는 것은 아니다.  
포함 형태로 확장된 것이라서 둘 다 써야 한다.  

그리고 당연히 gemini용 llm driver도 필요하다.

## 기본 사용법

자, 그럼 langgraph를 가져다가 멀티에이전트를 구성해보자.  
어떤 에이전트를 호출할지 선택하는 intro 에이전트, 그리고 실질적인 동작을 수행하는 help, smalltalk 에이전트 2가지를 구성해보겠다.  

일단 상태에 대한 정의가 필요하다.  

![](/images/img_7f62487db9a02653.png)
이건 에이전트 간 상태, 대화 공유 등에 사용될 것이다. 이게 핵심 구성 요소 중 하나다.  

그리고 먼저 intro 에이전트의 프롬프트를 구성했다.  

![](/images/img_ed3fa95c2febec54.png)
이 에이전트는 대화를 분석해서 어떤 에이전트에게 처리를 위임할지를 결정하는 간단한 에이전트다.  
그래서 해당 대화를 "분류"한 다음에, 어떤 에이전트로 보낼지를 결정한다.  

나머지 2개 에이전트는 intro가 던진 상태값을 토대로 최종 동작을 수행하는 에이전트다.  

![](/images/img_b80a0b860dc82206.png)

![](/images/img_0ac890efd36b99a8.png)
구현 자체는 매우 단순하다.  

그리고 저 3가지 에이전트를 가져다가 그래프 구조를 정의해야 한다.  

![](/images/img_018063a3493042f1.png)
각 에이전트를 "노드"로 등록하고, 어디에서 어떤 노드로 이동할 수 있는지를 그래프 구조로 선언한다.  
이렇게 정의된 흐름을 따라서 에이전트가 실행되는 것이다.  

<br>

![](/images/img_67c0a24b6f2868ce.png)
여기서 route_to_agent는 상태값의 intent 값을 토대로 어떤 노드를 보낼지를 골라준다.  

<br>

![](/images/img_f94aa5e3206241fd.png)
그리고 이 그래프를 토대로 "컴파일"을 실행하면, 최종적으로 호출할 수 있는 에이전트 flow가 완성된다.  

<br>

![](/images/img_565b9ee52b900ffb.png)
실제로 실행할 때는 컴파일된 app으로 invoke를 호출하면 된다. 그리고 최초 state를 주입해줘야 한다.  

자, 그러면 실행해보자.  

![](/images/img_9e31842a0cee2d12.png)
사소한 수다를 던지면 smalltalk 에이전트로 라우팅해주고  

<br>

![](/images/img_7bfbb76dcb8e37a6.png)
좀 제대로 된 질문을 던지면 help agent로 넘겨서 답해준다.  
근데 지식 수준이 구리긴 하더라. 저건 다 유명한 브랜드지, 가성비 나쁘고 비싼 브랜드다.
