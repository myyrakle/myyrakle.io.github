Kotlin Native는 Kotlin은 바로 실행 가능한 바이너리로 컴파일할 수 있게 해주는 툴체인이다.  
Kotlin 공식 그룹에서 함께 제공하는 오피셜 도구다.  

이걸 사용하면 코틀린 코드를 JVM을 사용하지 않는 독립적인 실행 파일로 컴파일할 수 있다.  

<br>

<br>

<br>

## 설치

Arch를 쓴다면 yay를 통해서 간편하게 설치할 수 있다.  

```javascript
yay -S kotlin-native-bin
```

그게 아니라면 코틀린 릴리즈 페이지에서 직접 받아다가 풀어쓰면 된다.  
[https://github.com/JetBrains/kotlin/releases/tag/v2.2.21](https://github.com/JetBrains/kotlin/releases/tag/v2.2.21)  

정상적으로 설치가 되었다면, 다음과 같이 전용 명령 도구를 쓸 수 있어야 한다.  

![](/images/img_dabf6720ae4cf32f.png)
이게 네이티브 컴파일러다.

<br>

## 컴파일해보기

코틀린 코드를 네이티브 실행파일로 컴파일해보자.  

다음과 같은 간단한 코드가 있다고 가정해보겠다.  

![](/images/img_e5b19d170eccd4da.png)

그러면, 원래는 코틀린 컴파일러를 통해 자바 바이트코드로 컴파일하고, java로 실행하는 것이 기본적인 사용 방법이다.  

```javascript
kotlinc Main.kt -include-runtime -d Main.jar
java -jar Main
```

![](/images/img_a146b8e7c43dc4e6.png)
당연하다. 코틀린의 출력은 JVM에 의존적이기 때문이다.  

하지만 kotlinc-native를 사용하면 JVM과 별개의 실행파일을 만들 수 있다.  
사용법은 다음과 같다.  

```javascript
kotlinc-native ./Main.kt -o main
```

![](/images/img_a17dfbdb51377b8d.png)

![](/images/img_bd1d0e89e7383371.png)
그럼 처음에는 거창하게 뭘 받으면서 컴파일을 한다.  
그래서 최초 컴파일은 상당히 오래 걸리는 편이다.  

그러고 나면  

![](/images/img_90dd089e6d0daaac.png)
.kexe라는 형식으로 실행파일이 생긴다. 그냥 실행하면 된다.  
끝이다.

## 한계

Kotlin Native의 가장 큰 문제는 java 라이브러리 구성요소들을 완전히 사용할 수 없다는 것이다.  
코드에 Java 종속적인 부분이 하나라도 있다면 무조건 컴파일에 실패한다.  

![](/images/img_77f0ee6649056423.png)

![](/images/img_e579c0707ebd460d.png)

내가 볼때 Kotlin Native를 아무도 쓰지 않는 이유는 이것 때문이다.  
애초에 코틀린을 쓰는 가장 큰 이유가 Java 환경을 그대로 쓸 수 있다는 건데, 그걸 잘라내면 누가 쓰나...  

그래서 대부분의 상황에서 현실적인 대안은 GraalVM일 것이다. 어차피 코틀린도 jar 파일을 뽑으니까, graalvm에 그걸 집어넣으면 AOT 컴파일이 된다. 사실 상위호환인 셈이다...  

```javascript
native-image -jar ./Main.jar -o ./main
```
[https://blog.naver.com/sssang97/224104258144](https://blog.naver.com/sssang97/224104258144)

## 성능 손실

그리고 내가 벤치마크 좀 돌려보니까, JVM으로 돌리는 것보다 현저하게 느리다.  
Java를 GraalVM으로 컴파일해도 성능 저하가 좀 보이는데, 이건 그거보다 더 심하다.  

JIT을 떠나서 그냥 최적화를 못하는 것 같다.
